## 1. 文档说明

### 1.1 文档目标

本文档是考研择校推荐系统数据采集模块的完整技术说明，覆盖以下内容：

1. 模块总体架构与设计原则。
2. 各爬虫的职责划分、接口契约、字段映射。
3. 数据来源能力分析（含接口探查结论与数据缺口说明）。
4. 标准运行流程与运维操作。
5. 数据治理策略：质量校验、降级方案、合规要求。
6. 验收标准与已知问题。

### 1.2 模块定位

数据采集模块是整个系统的数据入口，承担三类职责：

- 从外部数据源（掌上考研）采集结构化数据；
- 保留原始响应用于复核与降级；
- 输出标准化中间数据（CSV）供下游入库（S04/S07）消费。

本模块对应合并后的 S03 数据采集阶段，输出全部为中间文件，不直接写入业务库；写入业务库由 S04/S07 的 `src/data_pipeline/` 入库模块负责，以保证采集与入库职责分离、可独立测试。

### 1.3 设计原则

| 原则 | 说明 |
| --- | --- |
| 职责单一 | 每个爬虫只负责一个接口维度，互不依赖 |
| 原始留存 | 每次接口响应落盘，接口变更可重新解析 |
| 失败不静默 | 单条/单校失败记录日志并继续，不中断整体采集 |
| 可重复执行 | 同一批次可重跑，原始 JSON 覆盖写，解析 CSV 重新生成 |
| 频率可控 | 统一请求间隔，避免对数据源造成压力 |
| 合规优先 | 不采集用户隐私数据，不绕过登录鉴权采集会员数据 |

## 2. 总体架构

### 2.1 模块组成

```text
src/crawlers/
├── __init__.py                          # 包标识
├── base.py                              # 公共能力：HTTP 请求、重试、落盘、工具函数
├── kaoyan_school_list_crawler.py        # 爬虫一：院校花名册（schoolList）
├── kaoyan_plan_list_crawler.py          # 爬虫二：专业目录与招生计划（planList）
├── kaoyan_score_line_crawler.py         # 爬虫三：历年分数线（schoolScore）
└── kaoyan_level_rate_crawler.py         # 爬虫四：学科评估等级（schoolLevelRate）
```

### 2.2 数据流向

```text
掌上考研 API
      │  HTTPS POST
      ▼
┌─────────────┐    每页/每次响应     ┌──────────────────┐
│  base.py    │ ──────────────────▶ │  data/raw/<crawler>/<batch_id>/  （原始 JSON）
│ 请求+重试+   │                     │  保留接口原貌，供复核与降级重解析
│ 落盘公共层   │                     └──────────────────┘
└──────┬──────┘
       │  解析标准化
       ▼
┌──────────────────┐
│ data/processed/<domain>/<name>_<batch_id>.csv  （标准化中间数据）
│ 字段对齐业务表，供入库模块 S04/S07 消费
└──────────────────┘
       │
       ▼
入库模块（src/data_pipeline，S04/S07）→ MySQL 业务库
```

### 2.3 公共能力设计（base.py）

四个爬虫共享以下能力，避免重复实现：

| 能力 | 函数 | 说明 |
| --- | --- | --- |
| 统一请求头 | `DEFAULT_HEADERS` | 含 User-Agent、Origin、Referer，模拟浏览器请求 |
| HTTP 请求 | `post_api(path, payload)` | 封装 POST，含超时、重试、非 0000 code 检测 |
| 重试控制 | `MAX_RETRIES=2` | 超时或网络异常重试 2 次，非 JSON 响应直接失败 |
| 原始落盘 | `save_raw_json(dir, name, body)` | 以 UTF-8、缩进格式保存完整响应体 |
| 频率控制 | `sleep_briefly()` / `PAGE_DELAY=0.5s` | 统一请求间隔 |
| 工具函数 | `_to_int` / `major_category_code` | 类型转换与学科门类推导 |
| 日志 | 复用 `src.common.logger` 的 `crawler` logger | 日志统一进 `logs/crawler.log` |
| trace_id | 调用 `set_trace_id` | 每批次一个 trace_id，串起全链路日志 |

## 3. 数据来源能力分析

本节记录对掌上考研接口能力的实际探查结论，是数据缺口判断与降级方案设计的依据。

### 3.1 可用接口清单

| 接口 | 方法 | 业务含义 | 是否需登录 | 分页 | 采集状态 |
| --- | --- | --- | --- | --- | --- |
| `/pc/school/schoolList` | POST | 院校花名册 | 否 | 是 | 已采集 |
| `/pc/school/planList` | POST | 招生计划 + 专业目录 | 否 | 是 | 已采集 |
| `/pc/school/schoolScore` | POST | 历年分数线 | 否 | 否 | 已采集 |
| `/pc/school/planDetail` | POST | 单专业详情（参考书、复试科目） | 否 | 否 | 备用 |
| `/pc/school/schoolLevelRate` | POST | 学科评估等级 | 否 | 否 | 已采集 |

### 3.2 接口关键特性

1. **`schoolList`**：按 `province_id` 返回院校清单，分页参数 `page`/`limit`，返回 `total` 标识总数。重庆共 21 所。

2. **`planList`**：必须传 `school_id` 与 `year`。其 `recruit_number` 为「专业招生总数」，`remark` 明确标注「招生人数为专业招生总数，非方向人数」。

3. **`schoolScore`**：必须传 `year`，否则返回 0 条。响应含两类数据，由 `data_type` 区分：
   - `score_level`：学校按学科门类划定的校线；
   - `school_score`：具体院系专业的复试线。
   - 两类数据均含 `diff_total` 等字段，表示超出当年国家线的分差。

4. **`schoolLevelRate`**：按一级学科（4 位代码）返回评估等级（A+/A/A-/B+/B/B-/C+/C/C-），部分学科未参评，`rate` 为空。

### 3.3 数据缺口与补救

经探查，以下数据掌上考研**无法提供**：

| 缺口 | 业务影响 | 补救方案 |
| --- | --- | --- |
| **国家线**（`line_type=national`） | `score_lines` 表缺全国统一最低线 | 利用 `schoolScore.score_level` 的 `diff_total`（超国家线分差）反算国家线（`国家线 = 校线 - diff_total`）。经实测，重大 2024 哲学校线 340、diff 7，反推国家线 333，符合当年实际 |
| **推免人数 / 统考名额** | `enrollment_plans` 两列空 | `planList` 无此拆分；需从各校研究生院官网《推免拟录取名单》补充（S05） |
| **拟录取名单 / 报录比** | `admission_records` 表无数据 | 接口完全无此数据；需从官网《拟录取公示》PDF 提取。建议首版仅覆盖 3 所重点院校（重庆大学、西南大学、重庆邮电大学）作样例 |

### 3.4 合规说明

- 本模块仅采集**公开院校招生信息**，不涉及任何考生个人数据。
- `admission_records` 表设计已内置 `candidate_no_hash`（脱敏考生编号哈希）字段，未来若提取拟录取名单，仅保留脱敏后的统计信息，不存储明文考号。
- 不绕过登录鉴权采集会员专属数据（如 `queryAISchoolVip` 等 VIP 接口未采集）。

## 4. 爬虫详细设计

### 4.1 爬虫一：院校花名册（kaoyan_school_list_crawler）

| 项目 | 内容 |
| --- | --- |
| 接口 | `POST /pc/school/schoolList` |
| 请求参数 | `{"province_id": 50, "page": <n>, "limit": 20}` |
| 采集逻辑 | 自动分页至累计数等于 `total` |
| 原始落盘 | `data/raw/kaoyan_school_list/<batch_id>/page_<n>.json` |
| 解析落盘 | `data/processed/universities/university_candidates_<batch_id>.csv` |
| 目标表 | `universities` |
| 实际产出 | 21 条 |

**字段映射（→ universities）：**

| 接口字段 | 目标字段 | 转换 |
| --- | --- | --- |
| `school_id` | `candidate_school_id` | 原值 |
| `school_name` | `university_name` | 原值 |
| `province_name` | `province` | 原值（重庆） |
| `type_name` | `school_type` | 原值 |
| `type_school_name` | `school_org_type` | 原值 |
| is_985/is_211/syl/is_zihuaxian 组合 | `school_level` | 标志位组合（985/211/双一流/自划线/普通院校） |
| `recruit_number` | `recruit_number_reference` | 字符串转整型 |
| `major_number` | `major_number_reference` | 原值 |

**层次标志位规则**：接口标志位取值 1 表示「是」、其余表示「否」。组合规则：985 → 211 → 双一流 → 自划线，按存在顺序拼接；全部为否则判定为「普通院校」。

### 4.2 爬虫二：招生计划与专业目录（kaoyan_plan_list_crawler）

| 项目 | 内容 |
| --- | --- |
| 接口 | `POST /pc/school/planList` |
| 请求参数 | `{"school_id": <id>, "year": <y>, "page": <n>, "limit": 50}` |
| 采集逻辑 | 遍历 21 校 × 5 年，逐年分页至 `total` |
| 原始落盘 | `data/raw/kaoyan_plan_list/<batch_id>/school_<sid>_year_<y>_page_<n>.json` |
| 解析落盘 | `data/processed/plans/plan_list_<batch_id>.csv` |
| 目标表 | `departments`、`majors`、`enrollment_plans` |
| 实际产出 | 12712 条 |

一条 `planList` 记录拆分为三张表数据（入库阶段 S07 执行）：

**字段映射（→ departments）：**

| 接口字段 | 目标字段 |
| --- | --- |
| `depart_name` | `department_name` |
| 关联 `school_id` → `university_id` | 外键 |

**字段映射（→ majors）：**

| 接口字段 | 目标字段 | 转换 |
| --- | --- | --- |
| `special_code` | `major_code` | 保持字符串，避免前导零丢失 |
| `special_name` | `major_name` | 原值 |
| `special_code` 前 2 位 | `major_category` | 查 `config/major_category_dict.yaml` 推导 |
| `degree_type` | `degree_type` | 1→professional，2→academic |
| `recruit_type_name` | `study_mode` | 全日制→full_time，非全日制→part_time |
| `research_area` | `research_direction` | 原值 |
| `exam_subject` | `exam_subjects` | 清理 HTML `<br/>` |

**字段映射（→ enrollment_plans）：**

| 接口字段 | 目标字段 | 说明 |
| --- | --- | --- |
| `year` | `year` | 原值 |
| `recruit_number` | `plan_count` | 招生总人数 |
| — | `recommended_exemption_count` | **接口无数据，置 NULL** |
| — | `unified_exam_count` | **接口无数据，置 NULL** |

## 6. 数据治理

### 6.1 数据质量保障

| 环节 | 措施 |
| --- | --- |
| 采集容错 | 单校单年失败记入 `failed` 清单并继续，不中断 |
| 字段校验 | 数值字段安全转换（`_to_int`），非数字不报错返回 None |
| 前导零保护 | 专业代码全程按字符串处理，避免 `045300` → `45300` |
| HTML 清洗 | `exam_subject` 的 `<br/>` 统一替换为换行 |
| 标准化一致 | 学位类型、学习方式、分数线类型采用固定枚举值 |
| 空值处理 | 接口无数据的字段（如推免）明确置 NULL，不用占位符 |

### 6.2 降级方案

| 场景 | 降级措施 |
| --- | --- |
| 接口变更/字段缺失 | 保留原始 JSON，可重新编写解析逻辑，无需重新采集 |
| 接口限流或封禁 | 降低 `PAGE_DELAY` 频率；支持断点续采（按批次目录续跑） |
| 单校数据为空 | 自动跳过并记日志（如中科院国科大重庆学院部分年份无数据） |
| 采集中断 | 已落盘的原始 JSON 保留，重跑仅补采未完成部分 |
| 人工导入降级 | `universities` 入库支持人工 CSV 导入（S04） |

### 6.3 数据可追溯性

每条数据的来源可追溯：

- `crawler_runs` 表记录每个爬虫批次（批次号、接口、参数、抓取量、产出路径）；
- `source_documents` 表登记候选库和各接口/官网资料来源，S04 为 21 所候选学校登记 `school_list` 来源，后续 `majors`/`score_lines` 的 `source_id` 在 S07 关联；
- 原始 JSON 按批次归档在 `data/raw/<crawler>/<batch_id>/`，可定位到任意一条记录的原始响应。

## 7. 运维与监控

### 7.1 日志体系

- 统一使用 `src.common.logger`，`crawler` logger 输出至 `logs/crawler.log`（轮转 10MB×5）与控制台。
- 每条日志含 `trace_id`（批次级），便于跨文件追踪一次采集任务。
- ERROR 级别同步写入 `logs/error.log`。

### 7.2 健康检查要点

每次采集后应确认：

1. 摘要 `status` 为 success。
2. `fetched_count` 与预期量级吻合（招生计划应达万级）。
3. `failed` 清单为空或仅含已知空数据学校。
4. `data/processed/` 下 CSV 行数与 `fetched_count` 一致。

### 7.3 增量与重采

- **全量重采**：直接重跑，原始 JSON 覆盖、CSV 重新生成。
- **年度增量**：编辑 `YEARS` 常量仅保留新年份，跑完后入库阶段做 upsert（唯一键去重）。
- **新校扩展**：在 `CHONGQING_SCHOOL_IDS` 增加 school_id 即可，其余逻辑无需改动。

## 8. 与其他模块的协作

| 下游模块 | 消费内容 | 协作约定 |

## 9. 验收标准

1. 四个爬虫均可独立运行，退出码为 0。
2. 院校花名册采集不少于 21 条。
3. 招生计划采集不少于 10000 条（首批实测 12712）。
4. 历史分数线采集不少于 4000 条，且 `line_type` 正确区分 university/major（首批实测 4833）。
5. 学科评估采集不少于 500 条（首批实测 528）。
6. 原始 JSON 与解析 CSV 均落盘成功，行数一致。
7. 采集失败有明确日志，不静默失败。
8. 解析 CSV 字段与本文档第 4 章字段映射一致，可供入库模块直接消费。
