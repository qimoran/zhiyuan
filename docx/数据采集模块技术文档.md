## 1. 模块定位

数据采集模块对应项目 S03 阶段，负责从掌上考研公开接口采集重庆研招单位、招生计划、专业详情、分数线和学科评估数据。S03 只输出文件和日志，不直接写入 MySQL；候选学校入库由 S04 的 `src/data_pipeline/source_registry.py` 完成，专业、计划、分数线等明细由后续清洗入库模块处理。

本轮实现采用 S03 V2 方案：分块爬虫、聚合 JSON、统一调度、最终生成一个“方向级大合集 CSV”。最终 CSV 的粒度是一行一个 `plan_id + research_area + exam_subject_clean` 研究方向，每行同时保留学校、学院、专业、研究方向、考试科目、参考书、分数线和学科评估字段；接口没有返回的数据保持空值，不使用旧接口补假数据。

## 2. 设计原则

| 原则 | 说明 |
| --- | --- |
| 分块清晰 | 每类接口独立一个 crawler 文件，总入口只负责调度和整合 |
| 原始留存 | 每类接口保存一个聚合 JSON，便于复核和断点续跑 |
| 不重复采集 | 学校、计划、详情、方向、分数线和学科评估都有明确去重键 |
| 失败不静默 | 接口异常、限流、字段缺失和空数据都进入日志和 `crawl_summary.json` |
| 频率可控 | `planDetailV2` 默认单线程，真实请求间隔 2 秒，限流后退避 |
| 入库解耦 | S03 只生成文件，S04/S07 再根据 CSV 和来源记录入库 |
| 合规优先 | 只采集公开招生信息，不采集登录后数据和个人隐私数据 |


## 5. 采集范围

| 维度 | 范围 |
| --- | --- |
| 地域 | 重庆，`province_id=50` |
| 学校 | 掌上考研重庆筛选页返回的 21 所研招单位 |
| 年份 | 2024、2025、2026 |
| 招生计划 | `planListV2` 按学校 + 年份分页 |
| 专业详情 | `planDetailV2` 按 `plan_id` 去重采集 |
| 分数线 | `schoolScore` 按学校 + 年份采集 |
| 学科评估 | `schoolLevelRate` 按学校采集 |

## 6. 接口说明

| 分块 | 接口 | 请求参数核心字段 | 输出聚合 JSON |
| --- | --- | --- | --- |
| 院校花名册 | `/pc/school/schoolList` | `province_id=50`、`page`、`limit` | `school_list/school_list.json` |
| 招生计划 | `/pc/school/planListV2` | `school_id`、`year`、`page`、`limit` | `plan_list_v2/plan_list_v2.json` |
| 专业详情 | `/pc/school/planDetailV2` | `plan_id` | `plan_detail_v2/plan_detail_v2.json` |
| 分数线 | `/pc/school/schoolScore` | `school_id`、`year` | `score_line/score_line.json` |
| 学科评估 | `/pc/school/schoolLevelRate` | `school_id` | `level_rate/level_rate.json` |

说明：旧版计划列表和计划详情接口不再作为全量主流程使用。本项目以 `planListV2 + planDetailV2` 作为招生计划和专业详情的主来源。

## 7. 去重与续跑规则

| 数据域 | 去重键 | 说明 |
| --- | --- | --- |
| 学校 | `school_id` | 防止分页或接口重复返回同一学校 |
| 计划列表 | `(school_id, year, plan_id)` | 同校同年同计划只保留一条 |
| 专业详情 | `plan_id` | 已成功详情在 `--resume` 时跳过，失败详情可重新请求 |
| 最终方向行 | `(school_id, year, plan_id, research_area, exam_subject_clean)` | 保证最终 CSV 没有重复方向 |
| 分数线 | `(school_id, year, data_type, depart_id, code, name, degree_type)` | 用于分数线聚合去重 |
| 学科评估 | `(school_id, code, degree_type, rate)` | 用于学科评估聚合去重 |

同批次运行时会创建 `data/raw/kaoyan_v2/<batch_id>/.crawl.lock`。如果锁文件存在，说明同批次可能已有爬虫在运行，第二个进程会直接退出，防止 Docker 中多个进程同时写聚合 JSON 和最终 CSV。

## 8. 限流与异常处理

普通分页请求失败会记录异常并按配置重试。`planDetailV2` 是最容易触发限流的接口，当前策略为：

1. 默认单线程请求详情。
2. 未缓存详情请求之间默认等待 2 秒。
3. 如果返回 `code=0010` 且 `message=请求频繁,请稍后再试`，先每隔 2 秒快速重试。
4. 快速重试仍失败时等待 30 秒后再试。
5. 超过配置上限后，该 `plan_id` 详情记为 `source_status=error`，字段留空，整体采集继续运行。

每处理 10 条详情会在日志中输出一次进度，例如“已处理、成功、失败、剩余待成功”，方便从 `logs/` 目录观察后台爬虫状态。

## 9. 最终 CSV 口径

最终 CSV 输出路径：

```text
data/processed/kaoyan_v2_integrated/kaoyan_v2_integrated_<batch_id>.csv
```

最终 CSV 不再使用旧混合行类型字段。它只保留方向级计划行，一行代表一个研究方向。主要字段包括：

| 字段组 | 代表字段 |
| --- | --- |
| 批次与状态 | `batch_id`、`source_api`、`source_status`、`error_message` |
| 学校 | `school_id`、`school_name`、`province_name`、`city`、`school_level`、`type_name` |
| 学院与专业 | `year`、`plan_id`、`depart_id`、`depart_name`、`special_code`、`special_name`、`degree_type_name` |
| 方向与考试 | `research_area`、`exam_subject`、`exam_subject_clean`、`research_area_note`、`is_statistic_direction` |
| 参考书与详情 | `exam_book`、`exam_book_clean`、`exam_book_year`、`min_score`、`score_years_json`、`major_rate`、`doctoral_point` |
| 分数线合并 | `score_total`、`score_politics`、`score_english`、`score_special_one`、`score_special_two`、`score_diff_to_national` |
| 学科评估合并 | `subject_code`、`subject_name`、`level_rate`、`rate_sort`、`has_doctor` |

分数线只合并 `data_type=school_score` 且精确匹配 `school_id + year + depart_id + special_code + degree_type` 的记录。匹配不到或匹配不唯一时不强行填充。学科评估优先按 `level2_code` 匹配，匹配不到再回退到 `level1_code`。

## 11. 结果检查

每次采集后应检查：

1. `data/raw/kaoyan_v2/<batch_id>/` 下 5 个聚合 JSON 是否存在。
2. `crawl_summary.json` 中 `school_count`、`plan_rows`、`unique_plan_count`、`detail_success_count`、`detail_error_count` 是否与日志一致。
3. 最终 CSV 年份是否只包含 `2024/2025/2026`。
4. 全量批次学校数是否为 21。
5. 最终 CSV 是否不存在重复 `(school_id, year, plan_id, research_area, exam_subject_clean)`。
6. `exam_book_clean`、`research_area_note`、`min_score`、`score_years_json`、`major_rate`、`doctoral_point` 等字段有则写入，没有则留空。
7. 分数线没有被门类线误填到具体专业方向。
8. `source_registry.py --dry-run` 能从大合集 CSV 去重得到候选学校。

## 12. 数据缺口说明

掌上考研 V2 接口作为当前主数据源，但以下缺口仍需人工核验或后续补充：

| 缺口 | 当前处理 |
| --- | --- |
| 推免人数、统考名额 | `planListV2` 未提供明确拆分，CSV 留空或保留计划总数；人工核验记录中标注缺口 |
| 拟录取名单、报录比 | 掌上考研接口未提供，首版不自动提取，人工核验记录中标注为后续增强 |
| 部分学校或年份无计划/分数线 | 记录为空数据或错误，不伪造记录 |
| 参考书字段为空 | `planDetailV2` 有则写入 `exam_book_clean`，没有则留空 |
| 分数线无法精确匹配 | 不做门类线兜底，避免误导推荐结果 |

## 13. 与下游模块关系

| 下游模块 | 使用方式 |
| --- | --- |
| S04 候选库入库 | 从 `kaoyan_v2_integrated_*.csv` 中按 `school_id + school_name` 去重写入 `universities` |
| S05 人工核验与缺口登记 | 不再开发自动提取脚本，只记录官网抽查结果和缺失字段 |
| S06 数据清洗 | 以 S03 V2 大合集 CSV 为主输入，清理专业代码、研究方向、考试科目、参考书、分数线等字段 |
| S07 核心入库 | 将方向级大合集关联 `university_id`、`source_id` 后写入业务明细表 |
| S13 大数据分析 | 使用处理后的 CSV/MySQL/Hive 数据做趋势统计和可视化支撑 |

## 14. 验收标准

1. Docker 内 `python -m compileall /workspace/src` 通过。
2. 样本批次能生成 5 个聚合 JSON、`crawl_summary.json` 和 1 个最终 CSV。
3. 样本 CSV 只包含方向级计划行，不再有独立学校行、分数线行或学科评估行。
4. 全量批次学校数为 21，年份只包含 2024、2025、2026。
5. `planDetailV2` 成功数、失败数和 `crawl_summary.json` 一致。
6. 最终 CSV 无重复方向行。
7. `source_registry.py --dry-run` 可以从最终 CSV 生成候选学校入库摘要。
8. 日志能说明采集进度、失败原因、限流处理和最终输出路径。
