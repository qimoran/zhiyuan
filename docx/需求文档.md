## 2. 项目概述

### 2.1 项目背景

高考志愿填报需要综合考虑考生成绩、位次、科目类型、院校历年录取情况、专业热度、招生计划变化等因素。传统填报方式通常依赖人工查询和经验判断，存在数据分散、比较困难、风险识别不直观等问题。

本项目面向重庆地区高校志愿填报辅助场景，基于公开招生录取数据，构建从数据采集、数据清洗、数据存储、大数据分析、可视化展示到志愿推荐和 AI 建议报告生成的完整数据应用链路。系统通过分数位次换算和“冲稳保”推荐规则，为考生提供可解释的院校推荐结果和个性化填报建议。

### 2.2 项目目标

1. 建立高考历年录取数据采集与清洗流程，形成结构化数据资产。
2. 将清洗后的数据写入 MySQL，并同步到 HDFS/Hive，支撑查询与分析。
3. 使用 Spark、Pandas、NumPy 对录取趋势、位次波动、院校热度等进行统计分析。
4. 基于 Flask 和 ECharts 构建网页端查询与可视化展示系统。
5. 实现考生输入分数、省份、科目类型后，自动生成“冲、稳、保”三档院校推荐。
6. 调用大模型生成基于真实推荐数据的个性化志愿填报建议报告。
7. 形成可演示、可验收、可答辩的实训项目成果。

### 2.3 业务价值

1. 对考生和家长：降低信息查询成本，辅助理解历年录取趋势和填报风险。
2. 对项目组：完整实践 Python 爬虫、数据清洗、数据库、大数据平台、可视化和 AI 应用集成能力。
3. 对课程实训：形成具备真实业务流程和企业开发文档规范的数据应用项目。

### 2.4 项目范围

本期项目范围包括：

1. 公开招生录取数据采集。
2. 数据清洗、字段标准化和分数位次换算。
3. MySQL 数据库存储和 Hive/HDFS 数据同步。
4. Spark 统计分析脚本。
5. Web 查询页面和可视化看板。
6. 志愿推荐算法规则。
7. AI 建议报告生成。
8. 项目运行说明、测试说明和答辩材料支撑文档。

本期项目暂不包含：

1. 用户实名注册、短信验证、支付、会员等商业化功能。
2. 与教育考试院系统的内部接口对接。
3. 对志愿填报结果作录取承诺。
4. 复杂的多用户权限后台。
5. 移动端 App 原生开发。
6. 大规模线上高并发部署。

### 3.2 典型使用场景

#### 场景一：考生查询推荐院校

1. 考生进入系统首页。
2. 输入高考分数、省份、科目类型、批次等信息。
3. 系统根据一分一段表将分数换算为位次。
4. 系统根据历年院校录取位次和推荐规则生成“冲、稳、保”推荐列表。
5. 考生查看院校名称、所在地、院校类型、历年最低分、历年最低位次和推荐原因。

#### 场景二：查看院校录取趋势

1. 用户选择目标院校。
2. 系统展示该院校近几年最低录取分、最低位次、招生计划变化趋势。
3. 用户通过折线图、柱状图等方式观察分数波动和位次变化。
4. 系统提示是否存在明显“大小年”波动。

#### 场景三：生成 AI 志愿建议报告

1. 用户在推荐结果页面点击生成建议报告。
2. 系统整理考生分数、位次、科目类型、推荐院校列表、风险梯度等结构化数据。
3. 系统将结构化数据传入大模型。
4. 大模型根据系统提供的真实数据生成建议报告。
5. 用户查看报告中的梯度合理性、风险提示和专业选择建议。

## 4. 总体业务流程

### 4.1 数据处理流程

1. 确定数据源和采集字段。
2. 使用 Python 爬虫采集公开数据。
3. 将原始数据保存为 CSV、JSON 或数据库临时表。
4. 执行数据清洗，包括去重、缺失处理、字段格式统一、院校名称标准化。
5. 将清洗后的数据写入 MySQL。
6. 将数据同步到 HDFS/Hive。
7. 使用 Spark 进行统计分析。
8. 将分析结果回写 MySQL 或导出为前端可读取的数据表。
9. Web 端读取数据并进行查询、推荐和可视化展示。

### 4.2 推荐服务流程

1. 用户输入分数、省份、科目类型、批次等查询条件。
2. 系统校验输入合法性。
3. 系统查询对应年份和科目类型的一分一段表，换算考生位次。
4. 系统查询符合条件的院校历史录取数据。
5. 系统计算考生位次与院校历史录取位次之间的差值。
6. 系统结合多年位次均值、最近年份位次、波动情况划分推荐梯度。
7. 系统返回“冲、稳、保”推荐列表。
8. 用户可进一步查看院校详情或生成 AI 报告。

### 4.3 AI 报告生成流程

1. 系统收集本次推荐结果和必要上下文。
2. 后端生成结构化 prompt。
3. 大模型基于系统提供的数据生成报告。
4. 系统对模型输出进行基础安全校验和格式整理。
5. 前端展示报告，并标明报告仅作为志愿填报参考。

## 6. 功能需求

### 6.1 FR-01 数据采集

| 项目 | 内容 |
| --- | --- |
| 需求编号 | FR-01 |
| 需求名称 | 高考录取公开数据采集 |
| 优先级 | P0 |
| 使用角色 | 数据采集成员 |
| 需求描述 | 系统应支持从公开网站采集历年录取数据、一分一段表和院校基本信息，为后续清洗、存储和分析提供原始数据。 |

#### 功能说明

1. 支持采集院校投档线数据，字段包括招生年份、院校名称、省份、科目类型、批次、最低录取分、最低位次、招生计划数等。
2. 支持采集专业录取线数据，字段包括招生年份、院校名称、专业名称、科目类型、批次、最低分、最低位次等。
3. 支持采集一分一段表，字段包括年份、省份、科目类型、分数、同分人数、累计人数。
4. 支持采集院校基本信息，字段包括院校名称、院校代码、所在省份、办学层次、院校类型、公办民办属性等。
5. 采集过程应记录数据来源 URL、采集时间和采集状态。
6. 爬虫应设置合理的请求间隔，避免对公开网站造成压力。

#### 验收标准

1. 至少完成一个主要数据源的可运行采集脚本。
2. 采集结果能够保存为结构化文件或写入数据库临时表。
3. 原始数据字段含义清晰，能支撑后续清洗和入库。
4. 对采集失败、字段缺失、网页结构变化等情况有基础日志记录。

### 6.2 FR-02 数据清洗与标准化

| 项目 | 内容 |
| --- | --- |
| 需求编号 | FR-02 |
| 需求名称 | 数据清洗、去重和字段标准化 |
| 优先级 | P0 |
| 使用角色 | 数据处理成员 |
| 需求描述 | 系统应对采集数据进行清洗和标准化，保证不同来源、不同年份的数据口径一致。 |

#### 功能说明

1. 去除重复数据，重复判断维度至少包括年份、院校名称、科目类型、批次和专业名称。
2. 对缺失值进行处理，关键字段缺失的数据应进入异常数据表或异常文件。
3. 对院校名称进行标准化，统一简称、旧名称、全称等写法。
4. 对科目类型进行统一，例如文科、理科、物理类、历史类等。
5. 对分数、位次、招生计划数等数值字段进行类型转换。
6. 对跨年份比较需要的分数和位次进行换算。
7. 输出清洗报告，说明原始数据量、有效数据量、重复数据量、异常数据量。

#### 验收标准

1. 清洗脚本可重复执行。
2. 清洗后的数据字段命名统一、类型正确。
3. 异常数据不会直接进入核心业务表。
4. 能输出基础数据质量统计结果。

### 6.8 FR-08 志愿推荐

| 项目 | 内容 |
| --- | --- |
| 需求编号 | FR-08 |
| 需求名称 | “冲、稳、保”院校推荐 |
| 优先级 | P0 |
| 使用角色 | 考生、家长 |
| 需求描述 | 系统应根据考生分数、位次、科目类型和历年录取数据，生成“冲、稳、保”三档院校推荐列表。 |

#### 功能说明

1. 用户输入分数、省份、科目类型、批次等条件。
2. 系统换算考生位次。
3. 系统筛选符合条件的院校录取数据。
4. 系统计算考生位次与院校历史最低录取位次之间的差值。
5. 系统根据位次差、近年波动、招生计划变化等因素划分“冲、稳、保”。
6. 推荐结果应包含院校名称、所在地、院校类型、历年最低分、历年最低位次、推荐档位和推荐理由。
7. 推荐列表应支持按档位、院校类型、所在省份、最低位次等条件筛选或排序。
8. 当数据不足时，系统应提示“数据不足，仅供参考”，不能强行输出确定性结论。

#### 初始推荐规则

位次越靠前表示排名越好。设：

```text
位次差 = 院校近年参考最低录取位次 - 考生位次
```

当位次差为正数时，表示考生位次优于或接近该院校历史最低录取位次，风险相对较低。当位次差为负数时，表示考生位次低于该院校历史最低录取位次，存在冲刺风险。

建议初始规则如下，具体阈值可根据数据分布调整：

| 档位 | 初始判断条件 | 说明 |
| --- | --- | --- |
| 冲 | 位次差在 -3000 到 0 之间，或近年存在波动导致有机会进入 | 有一定风险，适合作为冲刺选择 |
| 稳 | 位次差在 0 到 5000 之间，且近年波动不大 | 录取概率相对适中 |
| 保 | 位次差大于 5000，且近年最低位次相对稳定 | 风险相对较低 |

说明：不同分数段、不同批次、不同科目类型的位次密度不同，最终阈值应配置化，不能固定写死在前端页面中。

#### 验收标准

1. 用户输入有效信息后，系统能够输出三档推荐列表。
2. 每条推荐结果都能看到数据依据。
3. 推荐结果不脱离真实历史数据。
4. 推荐规则在代码或配置中有清晰说明。
5. 数据不足时系统能够给出提示。

## 8. 接口需求

### 8.1 接口设计原则

1. 后端接口采用 Flask 提供 HTTP API。
2. 接口返回 JSON 格式数据。
3. 成功响应和失败响应格式保持统一。
4. 接口参数应进行合法性校验。
5. 接口错误信息应便于前端展示和后端定位。

### 8.2 统一响应格式

```json
{
  "code": 200,
  "message": "success",
  "data": {}
}
```

失败响应示例：

```json
{
  "code": 400,
  "message": "参数错误：分数不能为空",
  "data": null
}
```

## 9. 非功能需求

### 9.1 性能要求

1. 首页和主要查询页面在本地 Docker 环境中应能正常加载。
2. 推荐接口在常规数据量下响应时间建议控制在 3 秒以内。
3. 图表接口在常规数据量下响应时间建议控制在 3 秒以内。
4. 数据清洗和 Spark 分析任务允许离线执行，不要求实时完成。

### 9.2 可用性要求

1. 系统页面应具备清晰的输入框、按钮、筛选项和结果展示区域。
2. 用户输入错误时，应给出明确提示。
3. 数据为空时，应显示空状态提示，而不是页面报错。
4. 推荐结果应能让用户看懂原因，不只给出院校名称。

### 9.3 安全要求

1. 不在代码仓库中提交真实账号、密码、Token、API Key。
2. 数据库连接信息应通过配置文件或环境变量管理。
3. 后端接口应校验用户输入，避免 SQL 注入等基础安全问题。
4. 日志中不得输出敏感配置。
5. AI 报告不得生成违法、误导性或绝对化承诺内容。

### 9.4 可维护性要求

1. 项目目录结构应清晰区分爬虫、清洗、后端、前端、分析脚本和文档。
2. 核心配置应集中管理。
3. 关键业务逻辑应有必要注释。
4. 推荐规则应可配置，便于后续调整阈值。
5. README 应说明环境启动、依赖安装、数据导入和系统运行方式。

### 9.5 兼容性要求

1. Python 版本使用 3.12。
2. MySQL 使用 8.0。
3. Redis 使用 5.0.14。
4. Hadoop 使用 3.3.6。
5. Hive 使用 4.0.1。
6. Spark 使用 3.5.1。
7. 前端图表使用 ECharts。
8. Web 后端使用 Flask。

## 16. 约束与声明

1. 系统数据来源应为公开合法渠道。

5. 本项目为大数据专业实训项目，重点展示数据处理链路、分析能力和应用实现能力。
