## 2. 项目概述

### 2.1 项目背景

考研择校需要综合考虑目标地区、学校层次、专业方向、初试总分、单科成绩、历年复试线、拟录取分数、招生人数、推免比例、专业热度等因素。传统择校方式主要依赖人工查看各高校研究生院公告、论坛经验帖和个人判断，存在数据分散、格式不统一、历年对比困难、风险判断主观性较强等问题。

本项目面向重庆高校考研择校辅助场景，基于公开官方资料构建从“资料收集、来源登记、PDF/表格提取、数据清洗、数据存储、大数据分析、可视化展示、择校推荐、AI 建议报告”的完整数据应用链路。系统通过结构化分析历年复试线、拟录取分数和招生计划，为考研学生提供可解释的“冲刺、稳妥、保底”目标推荐。

### 2.2 项目目标

1. 建立重庆高校考研官方资料收集和来源登记流程，保证数据来源可追溯。
2. 对招生专业目录、复试线、拟录取名单、招生简章等资料进行半自动提取和人工校对，形成结构化数据资产。
3. 将标准化后的数据写入 MySQL，并同步到 HDFS/Hive，支撑在线查询和离线分析。
4. 使用 Spark、Pandas、NumPy 统计复试线变化、拟录取分数变化、招生人数变化、专业热度和竞争程度。
5. 基于 Flask 和 ECharts 构建网页端查询、推荐和可视化展示系统。
6. 实现用户输入初试总分、单科成绩、目标专业门类、学习方式、意向学校层次后，自动生成“冲刺、稳妥、保底”推荐列表。
7. 调用大模型或使用模板降级方式生成基于真实推荐数据的考研择校建议报告。
8. 形成可运行、可测试、可演示、可答辩的实训项目成果。

### 2.3 业务价值

1. 对考研学生：降低查找官方资料和横向对比学校专业的成本，辅助识别择校风险。
2. 对项目组：完整实践 Python 数据处理、PDF/Excel 解析、数据库、大数据平台、可视化和 AI 应用集成能力。
3. 对课程实训：形成具备真实业务流程、数据可追溯和企业文档规范的数据应用项目。

### 2.4 项目范围

本期项目范围包括：

1. 重庆高校研究生招生公开资料的人工收集和来源登记。
2. PDF、Excel、网页表格的半自动提取和人工校对。
3. 学校、学院、专业、复试线、拟录取名单、招生计划等核心数据清洗与标准化。
4. MySQL 数据库存储和 HDFS/Hive 数据同步。
5. Spark 统计分析脚本。
6. Web 查询页面、可视化看板和推荐页面。
7. 考研择校推荐规则。
8. AI 择校建议报告生成。
9. 项目运行说明、测试说明和答辩材料支撑文档。

本期项目暂不包含：

1. 用户实名注册、支付、会员、短信验证等商业化功能。
2. 与研招网或高校内部系统的非公开接口对接。
3. 对用户最终录取结果作任何承诺。
4. 复杂的多角色权限后台。
5. 移动端 App 原生开发。
6. 大规模线上高并发部署。
7. 覆盖全国所有高校和所有专业。
8. 对扫描版 PDF 的全自动 OCR 识别准确率承诺。

## 3. 用户角色与使用场景

### 3.1 用户角色

| 角色 | 角色说明 | 主要诉求 |
| --- | --- | --- |
| 考研学生 | 输入初试分数、单科成绩、目标专业等信息，查看择校推荐 | 快速了解重庆高校中适合报考的学校和专业风险梯度 |
| 项目组数据成员 | 负责收集官方资料、登记来源、整理原始文件 | 保证资料真实、完整、可追溯 |
| 项目组数据处理成员 | 负责 PDF/Excel 提取、清洗、入库、同步 | 保证字段标准统一、数据质量可控 |
| 项目组开发成员 | 负责接口、页面、推荐逻辑、可视化和 AI 报告 | 完成可运行系统和演示功能 |
| 数据维护人员 | 后续更新年份数据、检查数据质量 | 保证新增数据能按统一规则入库 |
| 指导教师或评审人员 | 查看项目成果、文档和演示效果 | 判断项目完整性、技术合理性和实训价值 |

### 3.2 典型使用场景

#### 场景一：学生查询考研择校推荐

1. 学生进入系统推荐页面。
2. 输入初试总分、政治、英语、业务课一、业务课二等成绩。
3. 选择目标专业门类、专业名称、学习方式、意向学校层次和目标年份。
4. 系统校验总分、单科分数和查询条件。
5. 系统判断用户是否满足国家线、院校线或专业复试线要求。
6. 系统根据历年复试线、拟录取最低分、拟录取平均分、招生人数、专业热度生成推荐结果。
7. 学生查看“冲刺、稳妥、保底”三类目标学校或专业，并查看推荐理由。

#### 场景二：查看目标专业历年趋势

1. 用户选择学校、学院、专业和年份范围。
2. 系统展示该专业近几年复试线、拟录取最低分、拟录取平均分、招生人数变化趋势。
3. 用户通过折线图、柱状图等方式观察分数线和招生规模变化。
4. 系统提示是否存在分数波动明显、招生人数缩减、数据年份不足等风险。

#### 场景三：数据成员登记官方 PDF 资料

1. 数据成员从高校研究生院官网下载招生目录、复试线、拟录取名单等资料。
2. 在来源索引表中登记学校名称、年份、文档类型、来源 URL、文件路径、采集时间和处理状态。
3. 将原始文件保存到项目规定目录。
4. 后续数据提取和校对结果与来源索引关联，保证每条核心数据可以追溯到原始文件。

#### 场景四：生成 AI 择校建议报告

1. 用户在推荐结果页面点击生成报告。
2. 系统整理用户成绩、目标专业、推荐列表、分数差、单科风险、数据年份等结构化信息。
3. 系统将结构化数据传入大模型，或在接口不可用时使用模板生成降级报告。
4. 报告展示择校梯度、风险提示、备考建议和调剂关注方向。
5. 报告末尾明确说明“本报告仅作为考研择校参考，不代表录取结果”。

## 4. 总体业务流程

### 4.1 数据处理流程

1. 确定重庆高校和专业范围。
2. 人工收集官方 PDF、Excel 和网页公告。
3. 登记资料来源索引。
4. 将原始文件保存到 `data/raw` 对应目录。
5. 使用 Python 工具进行半自动文本或表格提取。
6. 对提取结果进行人工校对和异常标记。
7. 对学校、学院、专业、年份、分数、招生人数等字段进行清洗和标准化。
8. 将标准化数据写入 MySQL。
9. 将清洗数据同步到 HDFS/Hive。
10. 使用 Spark 进行统计分析。
11. 将分析结果写回 MySQL 或导出为前端可读取的数据。
12. Web 端读取数据并进行查询、推荐和可视化展示。

### 4.2 推荐服务流程

1. 用户输入初试总分、单科成绩、目标专业、学习方式、学校层次等条件。
2. 系统校验输入合法性。
3. 系统查询对应专业门类和目标年份的国家线、院校线、专业复试线。
4. 系统判断用户是否存在总分或单科不过线风险。
5. 系统查询目标范围内的学校、学院、专业和历年录取数据。
6. 系统计算总分差、单科安全度、拟录取分差、招生人数稳定性和专业竞争度。
7. 系统根据推荐规则划分“冲刺、稳妥、保底”。
8. 系统返回推荐列表、推荐理由、风险提示和数据来源说明。
9. 用户可查看学校详情、专业趋势图或生成 AI 报告。

### 4.3 AI 报告生成流程

1. 系统收集本次推荐结果和必要上下文。
2. 后端生成结构化 prompt。
3. 大模型基于系统提供的数据生成报告。
4. 系统对模型输出进行基础安全校验和格式整理。
5. 如果大模型调用失败，系统使用模板生成降级报告。
6. 前端展示报告，并标明报告仅作为考研择校参考。

### 5.2 年份范围

1. 第一版优先收集近 3 年数据。
2. 如果某学校部分年份资料缺失，允许保留已有年份，但推荐结果必须提示“历史数据不足”。
3. 系统字段设计应支持后续扩展更多年份。

### 5.3 专业范围

2. 电子信息类。
3. 教育类。
4. 法学类。
5. 医学类。
6. 管理类。
7. 机械类。
8. 土木交通类。

### 7.9 FR-09 考研择校推荐

| 项目 | 内容 |
| --- | --- |
| 需求编号 | FR-09 |
| 需求名称 | “冲刺、稳妥、保底”目标推荐 |
| 优先级 | P0 |
| 使用角色 | 考研学生 |
| 需求描述 | 系统应根据用户初试分数、单科成绩、目标专业和历年数据，生成“冲刺、稳妥、保底”三类学校或专业推荐列表。 |

#### 功能说明

1. 用户输入初试总分、单科成绩、目标专业门类、专业名称、学习方式、意向学校层次等条件。
2. 系统先进行国家线、院校线、专业复试线评估。
3. 系统筛选符合条件的重庆高校和专业。
4. 系统计算总分差、单科安全度、拟录取分差、招生人数稳定性、专业热度等指标。
5. 系统根据配置化规则生成“冲刺、稳妥、保底”推荐结果。
6. 推荐结果应包含学校名称、学院名称、专业代码、专业名称、研究方向、学习方式、历年复试线、拟录取分数、招生人数、推荐档位和推荐理由。
7. 推荐列表应支持按档位、学校层次、专业热度、分数差等条件筛选或排序。
8. 对数据年份不足、缺少拟录取名单、报录比缺失的专业，应在推荐理由中提示。

#### 初始推荐规则

第一版采用规则模型，不做复杂机器学习。设：

```text
总分差 = 用户总分 - 近年参考复试线
拟录取分差 = 用户总分 - 近年拟录取平均分
单科分差 = 用户单科分数 - 对应单科线
```

推荐档位初始定义：

| 档位 | 初始判断思路 | 说明 |
| --- | --- | --- |
| 冲刺 | 总分高于复试线但低于或接近拟录取平均分，或学校层次较高、竞争较强 | 有一定风险，适合作为目标上限 |
| 稳妥 | 总分明显高于复试线，单科安全，接近或高于拟录取平均分 | 风险相对适中 |
| 保底 | 总分明显高于复试线和拟录取最低分，且招生人数稳定 | 风险相对较低 |

说明：不同专业门类、不同学校、不同年份的分数分布差异较大，最终阈值应配置化，不能固定写死在前端页面中。

#### 验收标准

1. 用户输入有效信息后，系统能够输出三档推荐列表。
2. 每条推荐结果都能看到数据依据。
3. 推荐结果不脱离真实历史数据。
4. 推荐规则在代码或配置中有清晰说明。
5. 页面和报告中不得出现“一定录取”“保证上岸”等绝对化承诺。

## 9. 接口需求

### 9.1 接口设计原则

1. 接口统一使用 JSON 格式。
2. 返回结构包含 `code`、`message`、`data`、`trace_id`。
3. 参数错误、数据为空、系统异常应使用统一错误码。
4. 接口不返回数据库密码、文件系统绝对敏感路径和大模型密钥。
5. 推荐和报告接口必须返回风险提示。

### 9.2 统一响应格式

```json
{
  "code": 0,
  "message": "success",
  "data": {},
  "trace_id": "202606151200000001"
}
```

错误响应示例：

```json
{
  "code": 40001,
  "message": "参数错误：初试总分不能为空",
  "data": null,
  "trace_id": "202606151200000002"
}
```

### 9.3 核心接口清单

| 接口路径 | 方法 | 功能 | 优先级 |
| --- | --- | --- | --- |

| `/api/major/list` | GET | 查询学校或专业列表 | P0 |

## 10. 非功能需求

### 10.1 性能要求

1. 普通查询接口在本地环境下响应时间建议控制在 3 秒以内。
2. 推荐接口在常规样例数据量下响应时间建议控制在 5 秒以内。
3. 图表接口应支持常见筛选条件下的快速查询。
4. Spark 分析任务允许离线运行，不要求实时返回。

### 10.2 可用性要求

1. 系统应提供清晰的启动说明和 Docker 环境检查脚本。
2. 页面输入项应有明确标签和错误提示。
3. 数据为空时应显示“暂无数据”或“该学校暂未收集对应年份数据”，不能显示空白页面。
4. 推荐结果应能让用户看懂原因，不只给出学校名称。

### 10.3 安全要求

1. 不采集非公开、需登录、涉及个人隐私的敏感数据。
2. 拟录取名单中如包含考生姓名、考生编号等敏感信息，展示和报告中应尽量脱敏或只做统计分析。
3. 不在代码仓库保存真实密码、Token、API Key。
4. AI 报告不得输出录取承诺、歧视性建议或明显误导性内容。

### 10.4 可维护性要求

1. 推荐规则应配置化，便于后续调整阈值。
2. 数据来源、清洗规则、异常数据应可追踪。
3. 项目目录结构应清晰区分代码、配置、数据、SQL、文档和脚本。
4. 核心模块应有基础单元测试或接口测试。
5. 代码应有必要中文注释，保留专业英文术语。

### 10.5 兼容性要求

1. 本地开发环境以 Windows + PowerShell + Docker Desktop 为主。
2. Python 版本以 3.12 为准。
3. MySQL 版本以 8.0 为准。
4. Redis 版本以 5.0.14 为准。
5. Hadoop、Hive、Spark 版本沿用当前 Docker 环境。

## 15. 风险与应对措施

| 风险 | 影响 | 应对措施 |
| --- | --- | --- |
| 官方 PDF 格式不统一 | 自动提取困难 | 采用半自动提取 + 人工校对，不承诺全自动识别 |
| 拟录取名单缺失或字段不完整 | 推荐准确性下降 | 使用复试线和招生计划作为基础指标，并提示数据不足 |
| 报录比不公开 | 竞争度分析不完整 | 报录比作为 P2 补充指标，不作为核心依赖 |
| 不同学校专业名称不一致 | 横向对比困难 | 建立专业名称和专业代码标准化规则 |
| 第三方院校库信息与官网不一致 | 可能导致学校名称、层次或招生人数错误 | 第三方数据只作为候选范围参考，核心字段必须通过学校官网、学院官网、研招网或考试院复核 |
| 数据收集工作量过大 | 影响开发进度 | 第一版限制学校、年份和专业范围 |
| 推荐结果被误解为录取承诺 | 存在误导风险 | 页面和报告中明确写明仅供参考 |
| 大模型接口不可用 | 报告功能受影响 | 提供模板降级报告 |
| Docker 环境差异 | 同学运行失败 | 使用项目内启动脚本和环境检查脚本 |

## 17. 约束与声明

1. 本系统只使用公开官方资料或项目组手动整理的数据，不采集非公开数据。
2. 第三方院校库只用于辅助确定候选范围和补充基础标签，不能替代高校官网、研招网或考试院公开信息。
3. 系统推荐结果仅作为考研择校参考，不代表最终录取结果。
4. 最终录取受当年招生计划、报考人数、复试规则、调剂政策、导师名额等因素影响。
5. 大模型报告只能基于系统提供的数据生成，不应编造学校、专业或录取信息。
6. 本项目为大数据专业实训项目，重点展示数据处理链路、分析能力和应用实现能力。
