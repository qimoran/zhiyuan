### 1.2 拆分原则

1. 每个步骤工作量控制在 3.5 到 4.5 人天，默认由 1 名程序员独立负责。
2. 每个步骤必须有明确输入、输出、测试方法和完成标准。
3. 步骤按项目依赖顺序排列，先完成基础设施、数据库、爬虫和数据闭环，再开发接口、推荐、前端、大数据和报告。
4. 能并行的步骤明确标注并行条件，减少 4 人小组等待时间。
5. 每个步骤都尽量可独立测试，不要求等到最后集成才发现问题。
6. 第一版优先保证可演示主链路：掌上考研数据采集、候选库入库、官方资料登记、样例数据入库、推荐接口、页面展示、AI 报告和大数据分析链路。
7. 所有命令示例默认使用 Windows PowerShell。

### 1.3 工作量说明

本文档中的“人天”表示 1 名开发人员 1 天的正常开发工作量。  
本项目建议拆分为 15 个开发步骤，每个步骤约 4 人天，总工作量约 60 人天，适合 4 人小组在 3 周左右完成。

### S03 掌上考研 V2 数据采集模块

| 项目 | 内容 |
| --- | --- |
| 工作量 | 4 人天 |
| 前置依赖 | S01 |
| 适合负责人 | 爬虫开发或数据采集成员 |
| 是否可并行 | 可与 S02 并行；入库功能依赖 S02 |
| 对应详细设计 | 第 6.1 节、第 12.2 节 |

#### 目标

实现掌上考研 V2 数据采集闭环：以 `https://www.kaoyan.cn/school-list/50-0-0` 为入口，调用公开接口采集重庆 21 所研招单位，并采集近三年（2024、2025、2026）的招生计划、专业详情、历年分数线和学科评估等级。S03 只负责“采集、原始留存、方向级大合集 CSV、日志和完整性记录”，不直接写入 MySQL；入库、来源登记和去重规则由 S04/S07 完成。

#### 任务清单

1. 创建 V2 分块爬虫：`school_list_crawler.py`、`plan_list_v2_crawler.py`、`plan_detail_v2_crawler.py`、`score_line_crawler.py`、`level_rate_crawler.py`。
2. 创建总入口 `src/crawlers/run_kaoyan_crawl.py`，统一调度分块爬虫并生成最终 CSV。
3. 固定采集重庆 `province_id=50` 的 21 所研招单位，试跑时可用 `--sample-schools 2` 只跑重庆大学和西南大学。
4. 默认采集近三年 `2024、2025、2026`，不再采集更早年份。
5. 使用 `planListV2` 采集计划列表，使用 `planDetailV2` 按 `plan_id` 采集详情，详情默认单线程、真实请求间隔 2 秒。
6. 原始响应改为聚合 JSON，统一保存到 `data/raw/kaoyan_v2/<batch_id>/` 下的五个接口域目录。
7. 最终只输出一个方向级大合集 CSV：`data/processed/kaoyan_v2_integrated/kaoyan_v2_integrated_<batch_id>.csv`。
8. 最终 CSV 一行代表一个 `plan_id + research_area + exam_subject_clean` 研究方向，不再输出单独的 `school`、`score_line`、`level_rate` 行。
9. 每行同时保留学校、学院、专业、研究方向、考试科目、参考书、分数线和学科评估字段；没有的数据留空，不使用旧接口补假数据。
10. 去重规则：学校按 `school_id`，计划按 `(school_id, year, plan_id)`，最终方向行按 `(school_id, year, plan_id, research_area, exam_subject_clean)`。
11. 分数线只按 `school_id + year + depart_id + special_code + degree_type` 精确匹配 `school_score`，多条或无匹配时留空，避免误把门类线写成专业线。
12. 学科评估优先按 `school_id + level2_code` 匹配，匹配不到再回退到 `level1_code`。
13. 支持 `--batch-id`、`--years`、`--sample-schools`、`--full`、`--resume`、`--detail-delay`、`--detail-backoff-rounds` 参数。
14. 同批次运行使用 `data/raw/kaoyan_v2/<batch_id>/.crawl.lock` 加锁，防止多个 Docker 进程同时覆盖 JSON/CSV。
15. 对接口失败、超时、限流、字段缺失、单校单年无数据进行日志记录，不静默失败；`planDetailV2` 遇到 `code=0010` 且提示“请求频繁,请稍后再试”时先短间隔重试，再按 30 秒退避。
16. 输出 `crawl_summary.json`，记录 `plan_rows`、`unique_plan_count`、`detail_success_count`、`detail_error_count`、`score_matched_count`、`level_rate_matched_count` 等完整性统计。

1. `src/crawlers/kaoyan_v2_common.py`
2. `src/crawlers/school_list_crawler.py`
3. `src/crawlers/plan_list_v2_crawler.py`
4. `src/crawlers/plan_detail_v2_crawler.py`
5. `src/crawlers/score_line_crawler.py`
6. `src/crawlers/level_rate_crawler.py`
7. `src/crawlers/run_kaoyan_crawl.py`
8. `data/raw/kaoyan_v2/<batch_id>/` 聚合 JSON
9. `data/processed/kaoyan_v2_integrated/kaoyan_v2_integrated_<batch_id>.csv` 方向级大合集 CSV

## 9. 风险与处理建议

| 风险 | 影响 | 处理建议 |
| --- | --- | --- |
| 掌上考研接口变更 | 爬虫无法抓取候选库或计划详情 | 保存最近一次聚合 JSON，保留人工导入 CSV 降级方案 |
| 掌上考研部分学校接口无数据 | 招生计划、分数线或学科评估覆盖不完整 | S03 记录缺口，S05 人工核验说明缺口，S06/S07 写入数据质量问题 |
| 官方 PDF 格式复杂 | 自动提取成本高 | 首版不做自动提取，只进行人工核验和缺口登记 |
| 样例数据不足 | 推荐结果不够丰富 | 第一版先保证 3 所学校、3 个专业门类可完整演示 |
| Hive 或 Spark 启动慢 | 影响演示 | 提前运行环境检查脚本，准备 Spark 输出截图或日志 |
| 大模型接口不可用 | 报告生成失败 | 模板报告作为默认可用方案 |
| 同学开发环境不一致 | 协作成本高 | 所有命令以 Docker 和 PowerShell 为准，README 写清楚 |
