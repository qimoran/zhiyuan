# 掌上考研扩展爬虫设计方案（S03b）

| 项目 | 内容 |
| --- | --- |
| 文档名称 | 掌上考研扩展爬虫设计方案 |
| 所属项目 | 重庆高校考研择校推荐系统 |
| 文档版本 | V1.0 |
| 编写日期 | 2026-06-15 |
| 依据文档 | `docx/开发步骤文档.md` V2.0（S03 节）、`sql/mysql/001_create_tables.sql` |
| 适用对象 | 爬虫与数据采集成员（同学 A）、数据清洗成员（同学 B） |
| 文档目标 | 在原 S03 院校列表爬虫基础上，扩展 3 个爬虫获取招生计划、历年分数线、学科评估数据，补足推荐系统所需的核心业务数据 |

## 1. 背景与目标

### 1.1 为什么要扩展

原 `开发步骤文档.md` 的 S03 只从掌上考研抓取重庆 21 所研招单位的**院校花名册**（`school/schoolList`），字段仅包含学校名、类型、层次、参考招生人数等。后续推荐系统真正需要的**专业目录、招生计划、历年分数线**在原设计中全部依赖 S05 人工扒取 21 所学校研究生院官网的 PDF、Excel。

经实际探查掌上考研前端接口发现：**专业目录、招生计划、历年分数线、学科评估等级都可以通过接口直接获取，无需登录、无需会员**。如果继续只抓花名册，会导致：

1. 推荐模块（S09/S10）缺少真实分数线和招生计划，只能靠少量样例数据演示；
2. 21 所学校官网 PDF 格式各异，人工提取错误率高，4 人 3 周难以完成；
3. 数据量不足，Hive/Spark（S13）和图表（S12）可分析的内容单薄。

因此新增本方案，在不改动原 S03 院校列表爬虫的前提下，扩展 3 个爬虫。

### 1.2 设计原则

1. 复用 S03 已建立的爬虫架构：`requests` + 自动分页 + 原始 JSON 落盘 + 统一日志 + 失败不静默。
2. 每个爬虫的输出字段与 `001_create_tables.sql` 中的表结构对齐，便于 S07 直接入库。
3. 所有原始响应保存到 `data/raw/`，解析结果保存到 `data/processed/`，保留降级与复核能力。
4. 控制请求频率，避免对接口造成压力或触发限流。

## 2. 接口探查结论

### 2.1 可用接口清单

| 接口 | 方法 | 业务含义 | 是否需登录 | 是否分页 | 探查结果 |
| --- | --- | --- | --- | --- | --- |
| `/pc/school/schoolList` | POST | 院校花名册 | 否 | 是 | 已实现（S03） |
| `/pc/school/planList` | POST | 招生计划 + 专业目录 | 否 | 是 | **可用**，2026 年重庆大学 450 条 |
| `/pc/school/schoolScore` | POST | 历年分数线（校线 + 专业线） | 否 | 否（按年返回全量） | **可用**，需传 year |
| `/pc/school/planDetail` | POST | 单专业详情（参考书、复试科目） | 否 | 否 | 可用，作为补充 |
| `/pc/school/schoolLevelRate` | POST | 学科评估等级（A+/A/B+…） | 否 | 否 | **可用**，重庆大学 81 条 |

> 说明：探查时尝试的 `/pc/school/majorList`、`/pc/school/schoolDetail` 等路径返回 404，实际专业目录数据由 `planList` 承载。`recruitSchoolList` 返回的是全国招生简章文章列表，不是拟录取名单。

### 2.2 关键发现（重要，影响后续设计）

1. **`planList` 不含推免人数**。其 `recruit_number` 字段是「专业招生总数」（接口 `remark` 明确说明「招生人数为专业招生总数，非方向人数」）。因此 `enrollment_plans.recommended_exemption_count`（推免人数）和 `unified_exam_count`（统考名额）**无法从该接口获得**，入库时置空，后续可由官网 PDF 补充。

2. **`schoolScore` 必须传 `year` 参数**，不传则返回 0 条。可获取年份范围为 2020—2026 共 7 年。

3. **`schoolScore` 返回两类数据**，由 `data_type` 区分：
   - `score_level`：学校按学科门类划的校线（如「01 哲学 总分 340」），对应 `line_type = university`；
   - `school_score`：具体院系专业的复试线（如「经济与工商管理学院 020200 应用经济学 总分 380」），对应 `line_type = major`。
   - 两种数据都自带 `diff_total`、`diff_politics` 等字段，表示**超出国家线的分差**，对推荐算法非常有价值。

4. **`schoolLevelRate` 是一级学科维度**（如 `0802 机械工程 = A-`），不是具体专业。需通过专业代码前 4 位（一级学科代码）与专业关联。

5. **拟录取名单、报录比接口未找到**。`admission_records` 表的数据仍需 S05 从官网 PDF 提取，本方案不覆盖。

### 2.3 数据量与请求量评估

以重庆 21 所研招单位为采集范围：

| 爬虫 | 采集维度 | 预估数据量 | 预估请求数 |
| --- | --- | --- | --- |
| planList | 21 校 × 5 年（2022—2026） | 约 8000—15000 条专业方向 | 约 1000 次（每校每年约 10 页） |
| schoolScore | 21 校 × 7 年（2020—2026） | 约 1500—2000 条分数线 | 147 次（每年 1 次请求） |
| schoolLevelRate | 21 校 | 约 1000—1500 条学科评级 | 21 次 |

> 说明：planList 按年逐年分页采集，每页 limit 取 50；schoolScore 单次请求返回某校某年全部分数线，无需分页；schoolLevelRate 单次请求返回某校全部评级。总请求数约 1200 次，按 0.5 秒间隔估算，全量采集约 10—15 分钟。

## 3. 爬虫一：招生计划与专业目录（planList）

### 3.1 基本信息

| 项目 | 内容 |
| --- | --- |
| 模块路径 | `src/crawlers/kaoyan_plan_list_crawler.py` |
| 接口地址 | `https://api.kaoyan.cn/pc/school/planList` |
| 请求方式 | POST，`Content-Type: application/json` |
| 请求参数 | `{"school_id": <院校ID>, "year": <年份>, "page": <页码>, "limit": 50}` |
| 采集来源 | 遍历 `universities.candidate_school_id`（S03 已入库的 21 个 school_id） |
| 采集年份 | 2022、2023、2024、2025、2026（可配置） |
| 原始落盘 | `data/raw/kaoyan_plan_list/<batch_id>/school_<school_id>_year_<year>_page_<n>.json` |
| 解析落盘 | `data/processed/plans/plan_list_<batch_id>.csv` |

### 3.2 原始字段（planList 返回，共 17 个）

| 原始字段 | 含义 | 示例值 |
| --- | --- | --- |
| `plan_id` | 招生计划记录 ID | 1331791 |
| `spe_id` | 专业方向 ID | 20134 |
| `year` | 招生年份 | 2026 |
| `school_id` | 院校 ID（请求时传入，回填） | 252 |
| `depart_id` | 学院 ID | 15739 |
| `depart_name` | 学院名称 | 外国语学院 |
| `special_code` | 专业代码 | 045300 |
| `special_name` | 专业名称 | 国际中文教育 |
| `degree_type` | 学位类型代码（1=专硕，2=学硕） | 1 |
| `degree_type_name` | 学位类型名称 | 专业型硕士 |
| `recruit_type_name` | 学习方式 | 全日制 |
| `exam_class` | 考试方式代码 | 1 |
| `exam_class_name` | 考试方式名称 | 统考 |
| `research_area` | 研究方向 | （01）外国语言学及应用语言学 |
| `exam_subject` | 初试科目（含 HTML `<br/>`） | ①(101)思想政治理论<br/>②(201)英语（一）… |
| `recruit_number` | 招生人数（专业总数） | 22 |
| `remark` | 备注 | 招生人数为专业招生总数，非方向人数 |

### 3.3 字段映射（planList → departments / majors / enrollment_plans）

一条 planList 记录会被拆成三张表的数据：先 upsert `departments`，再 upsert `majors`，最后写 `enrollment_plans`。

**departments 表映射：**

| 目标字段 | 来源 | 说明 |
| --- | --- | --- |
| `university_id` | 按 `school_id` 查 `universities` | 外键 |
| `department_name` | `depart_name` | 学院名称 |
| `source_id` | 本爬虫批次登记的来源 ID | 统一来源 |

> 唯一键 `uk_departments_university_name (university_id, department_name)` 保证同一学校同一学院不重复。

**majors 表映射：**

| 目标字段 | 来源 | 转换说明 |
| --- | --- | --- |
| `university_id` | 按 `school_id` 查表 | 外键 |
| `department_id` | 上一步 upsert 得到的学院 ID | 外键 |
| `major_code` | `special_code` | 保持字符串，避免前导零丢失（如 045300） |
| `major_name` | `special_name` | 专业名称 |
| `major_category` | `special_code` 前 2 位查门类字典 | 如 04→教育，08→工学；字典见 §6 |
| `degree_type` | `degree_type` | 1→`professional`（专硕），2→`academic`（学硕） |
| `study_mode` | `recruit_type_name` | 全日制→`full_time`，非全日制→`part_time` |
| `research_direction` | `research_area` | 研究方向 |
| `exam_subjects` | `exam_subject` | 清洗掉 `<br/>` 后存入 |
| `source_id` | 本批次来源 ID | 统一来源 |

> 唯一键 `uk_majors_unique (university_id, department_id, major_code, study_mode, research_direction)` 保证同一学校同学院同专业同方向不重复。

**enrollment_plans 表映射：**

| 目标字段 | 来源 | 说明 |
| --- | --- | --- |
| `year` | `year` | 招生年份 |
| `university_id` | 按 `school_id` 查表 | 外键 |
| `department_id` | 学院 ID | 外键 |
| `major_id` | 专业 ID | 外键 |
| `plan_count` | `recruit_number` | 招生总人数 |
| `recommended_exemption_count` | — | **接口无此数据，置 NULL** |
| `unified_exam_count` | — | **接口无此数据，置 NULL** |
| `source_id` | 本批次来源 ID | 统一来源 |

> 唯一键 `uk_enrollment_plans_unique (year, university_id, major_id)` 保证同年同校同专业不重复。

### 3.4 处理流程

```text
读取 universities 表 21 个 candidate_school_id
for school_id in 学校列表:
    for year in [2022, 2023, 2024, 2025, 2026]:
        page = 1
        while 累计条数 < total:
            POST planList(school_id, year, page, limit=50)
            保存原始 JSON
            累计记录
            page += 1
            sleep(0.5)
解析所有记录 → 写 CSV
```

## 4. 爬虫二：历年分数线（schoolScore）

### 4.1 基本信息

| 项目 | 内容 |
| --- | --- |
| 模块路径 | `src/crawlers/kaoyan_score_line_crawler.py` |
| 接口地址 | `https://api.kaoyan.cn/pc/school/schoolScore` |
| 请求方式 | POST |
| 请求参数 | `{"school_id": <院校ID>, "year": <年份>}` |
| 采集年份 | 2020—2026（共 7 年，可配置） |
| 原始落盘 | `data/raw/kaoyan_score_line/<batch_id>/school_<school_id>_year_<year>.json` |
| 解析落盘 | `data/processed/score_lines/score_lines_<batch_id>.csv` |

### 4.2 原始字段（schoolScore 返回，共 25 个）

| 原始字段 | 含义 |
| --- | --- |
| `data_type` | 数据类型：`score_level`（校线）/ `school_score`（专业线） |
| `year` | 年份 |
| `school_id` / `school_name` | 院校 ID / 名称 |
| `depart_id` / `depart_name` | 学院 ID / 名称（score_level 为 0/不区分院系） |
| `code` | 学科代码：score_level 为门类码（01），school_score 为专业码（020200） |
| `name` | 学科/专业名称 |
| `degree_type` | 学位类型（2=学硕等） |
| `total` | 总分线 |
| `politics` / `english` | 政治 / 英语线 |
| `special_one` / `special_two` | 业务课一 / 业务课二线 |
| `diff_total` 等 | 超出国家线的分差（推荐加分项） |
| `note` / `special_remark` | 备注 |

### 4.3 字段映射（schoolScore → score_lines）

| 目标字段 | 来源 | 转换说明 |
| --- | --- | --- |
| `year` | `year` | 年份 |
| `line_type` | `data_type` | `score_level`→`university`，`school_score`→`major` |
| `university_id` | 按 `school_id` 查表 | 外键 |
| `department_id` | 按 `depart_id`/`depart_name` 关联 | score_level 无院系则置 NULL |
| `major_id` | `school_score` 按 `(university, department, code)` 关联 majors | score_level 置 NULL |
| `major_category` | score_level 取 `name`；school_score 由 `code` 前 2 位推导 | 专业门类 |
| `total_score_line` | `total` | 总分线 |
| `politics_line` | `politics` | 政治线 |
| `english_line` | `english` | 英语线 |
| `subject_one_line` | `special_one` | 业务课一线 |
| `subject_two_line` | `special_two` | 业务课二线 |
| `source_id` | 本批次来源 ID | 统一来源 |

> **可选增强**：`diff_total`（超国家线分差）对推荐有用，但 `score_lines` 表无对应字段。两种处理方式：① 暂时拼入备注；② 建议给 `score_lines` 增加字段 `score_diff_to_national INT NULL`。建议采用方式②，改动小且收益明显，需在 S02 同步。

> 唯一键 `uk_score_lines_unique (year, line_type, university_id, major_id, major_category)` 保证不重复。

### 4.4 处理流程

```text
读取 universities 表 21 个 candidate_school_id
for school_id in 学校列表:
    for year in range(2020, 2027):
        POST schoolScore(school_id, year)   # 单次返回该校该年全部
        保存原始 JSON
        sleep(0.5)
按 data_type 分类解析 → 写 CSV
```

## 5. 爬虫三：学科评估等级（schoolLevelRate）

### 5.1 基本信息

| 项目 | 内容 |
| --- | --- |
| 模块路径 | `src/crawlers/kaoyan_level_rate_crawler.py` |
| 接口地址 | `https://api.kaoyan.cn/pc/school/schoolLevelRate` |
| 请求方式 | POST |
| 请求参数 | `{"school_id": <院校ID>}` |
| 原始落盘 | `data/raw/kaoyan_level_rate/<batch_id>/school_<school_id>.json` |
| 解析落盘 | `data/processed/level_rates/level_rates_<batch_id>.csv` |

### 5.2 原始字段

| 原始字段 | 含义 |
| --- | --- |
| `school_id` / `school_name` | 院校 ID / 名称 |
| `code` | 一级学科代码（4 位，如 0802） |
| `code_name` | 一级学科名称（如 机械工程） |
| `degree_type` | 学位类型 |
| `rate` | 评估等级（A+、A、A-、B+、B、B-、C+、C、C-，可能为空） |
| `rate_sort` | 等级排序值（数值越小越靠前） |
| `has_doctor` | 是否有博士点（1/0） |

### 5.3 字段映射与落表建议

学科评估是一级学科维度，现有表无完全对应字段。建议采用**轻量新表**方式，避免改动 majors 表结构：

**新增表 `subject_level_rates`（建议）：**

| 字段 | 类型 | 来源 |
| --- | --- | --- |
| `id` | BIGINT PK | 自增 |
| `university_id` | BIGINT | 按 school_id 关联 |
| `subject_code` | VARCHAR(20) | `code`（一级学科代码） |
| `subject_name` | VARCHAR(150) | `code_name` |
| `level_rate` | VARCHAR(10) | `rate` |
| `rate_sort` | INT | `rate_sort` |
| `has_doctor` | TINYINT | `has_doctor` |
| 唯一键 | | `(university_id, subject_code)` |

推荐模块（S10）使用时，按专业代码前 4 位（`major_code` 的前 4 位即一级学科）关联 `subject_level_rates`，作为「学科实力」加分项。

## 10. 风险与降级

| 风险 | 影响 | 处理建议 |
| --- | --- | --- |
| 接口限流或封 IP | 采集中断 | 控制 0.5 秒间隔，失败重试 2 次；保留每页原始 JSON，支持断点续采 |
| 接口字段变更 | 解析失败 | 解析容错，字段缺失记日志不中断；保留原始 JSON 可重新解析 |
| planList 无推免数据 | enrollment_plans 部分字段空 | 置 NULL，由 S05 官网 PDF 补充，不阻塞主链路 |
| 学校 id 在 universities 未入库 | 关联失败 | 采集前校验 candidate_school_id 已存在，缺失则跳过并记录 |
| 拟录取数据缺失 | admission_records 为空 | 维持原 S05 PDF 方案；推荐算法以分数线 + 招生计划 + 学科评级为主，拟录取作为补充 |

## 11. 验收标准

1. planList 爬虫：能抓取 21 校 × 多年招生计划，重庆大学 2026 年不少于 450 条。
2. schoolScore 爬虫：能抓取 21 校 × 7 年分数线，且 `line_type` 正确区分 university / major。
3. schoolLevelRate 爬虫：能抓取 21 校学科评估等级，重庆大学不少于 80 条。
4. 三个爬虫的原始 JSON 与解析 CSV 均落盘成功。
5. 失败时有明确日志，不静默失败。
6. 解析 CSV 字段名与本文档字段映射一致，可供 S07 直接入库。
