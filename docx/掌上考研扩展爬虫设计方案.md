# 掌上考研 V2 数据采集设计方案（S03）

| 项目 | 内容 |
| --- | --- |
| 文档名称 | 掌上考研 V2 数据采集设计方案 |
| 所属项目 | 重庆高校考研择校推荐系统 |
| 文档版本 | V2.0 |
| 更新日期 | 2026-06-16 |
| 适用对象 | 爬虫与数据采集成员、数据清洗成员、后端入库成员 |
| 文档目标 | 说明掌上考研 V2 分块爬虫的接口范围、落盘方式、去重规则、最终 CSV 口径和运行验收方式 |

## 1. 背景与目标

项目最初只计划从掌上考研重庆院校库采集 21 所研招单位候选清单。实际探查后发现，掌上考研公开接口还可以返回招生计划、专业详情、分数线和学科评估信息。为了减少人工整理压力，并给推荐、图表和大数据分析提供更完整的结构化数据，S03 已升级为 V2 分块采集方案。

V2 的目标不是把所有接口数据分别保存成很多零散 CSV，而是：

1. 分块采集，职责清楚。
2. 原始数据按接口域聚合成少量 JSON。
3. 最终生成一个方向级大合集 CSV。
4. 后续 S04/S07 从这个 CSV 继续入库和清洗。

最终 CSV 的粒度固定为：一行 = 一个 `plan_id + research_area + exam_subject_clean` 研究方向。学校、学院、专业、研究方向、考试科目、参考书、分数线、学科评估尽量合并到同一行；没有的数据留空。

## 2. 接口范围

| 数据域 | 接口 | 说明 |
| --- | --- | --- |
| 院校花名册 | `/pc/school/schoolList` | 固定 `province_id=50`，获取重庆 21 所研招单位 |
| 招生计划 | `/pc/school/planListV2` | 按学校 + 年份分页采集计划列表 |
| 专业详情 | `/pc/school/planDetailV2` | 按 `plan_id` 获取参考书、方向明细、近年分数等详情 |
| 历年分数线 | `/pc/school/schoolScore` | 按学校 + 年份采集校线和专业线 |
| 学科评估 | `/pc/school/schoolLevelRate` | 按学校采集学科评估等级 |

旧版计划列表和计划详情接口不再作为全量主流程使用。V2 主流程只使用 `planListV2 + planDetailV2` 作为招生计划和详情来源。

## 3. 代码结构

| 文件 | 职责 |
| --- | --- |
| `src/crawlers/kaoyan_v2_common.py` | 公共配置、请求、路径、锁、JSON 读写和字段清洗 |

| `src/crawlers/level_rate_crawler.py` | 采集学科评估 |
| `src/crawlers/run_kaoyan_crawl.py` | 总入口，调度分块爬虫并生成大合集 CSV |

## 4. 原始数据保存

每个批次保存到 `data/raw/kaoyan_v2/<batch_id>/`：

```text
data/raw/kaoyan_v2/<batch_id>/
├── .crawl.lock
├── school_list/school_list.json
├── plan_list_v2/plan_list_v2.json
├── plan_detail_v2/plan_detail_v2.json
├── score_line/score_line.json
├── level_rate/level_rate.json
└── crawl_summary.json
```

说明：

1. 每个接口域只保存一个聚合 JSON，避免产生大量零散文件。
2. `--resume` 会读取已有聚合 JSON，已成功的数据跳过，失败或缺失的数据继续尝试。
3. `.crawl.lock` 用于同批次运行锁，避免多个进程同时写同一批次文件。

## 5. 最终 CSV 设计

输出路径：

```text
data/processed/kaoyan_v2_integrated/kaoyan_v2_integrated_<batch_id>.csv
```

最终 CSV 不再输出 `school`、`score_line`、`level_rate` 这类独立行，也不再使用旧混合行类型字段。它只输出招生计划研究方向行。

主要字段：

| 字段类型 | 字段示例 |
| --- | --- |
| 学校字段 | `school_id`、`school_name`、`province_name`、`school_level`、`type_name` |
| 计划字段 | `year`、`plan_id`、`depart_id`、`depart_name`、`special_code`、`special_name` |
| 方向字段 | `research_area`、`exam_subject`、`exam_subject_clean`、`research_area_note`、`is_statistic_direction` |
| 参考书字段 | `exam_book`、`exam_book_clean`、`exam_book_year` |
| 详情字段 | `min_score`、`score_years_json`、`major_rate`、`doctoral_point` |
| 分数线字段 | `score_total`、`score_politics`、`score_english`、`score_special_one`、`score_special_two`、`score_diff_to_national` |
| 学科评估字段 | `subject_code`、`subject_name`、`level_rate`、`rate_sort`、`has_doctor` |

## 6. 合并规则

### 6.1 招生计划与详情

`planListV2` 提供计划基础字段，`planDetailV2` 按 `plan_id` 补充详情。详情中的 `research_area_data` 会展开为多行，因此一个 `plan_id` 可能对应多个研究方向行。

最终方向行去重键：

```text
(school_id, year, plan_id, research_area, exam_subject_clean)
```

### 6.2 分数线

分数线只合并精确匹配的专业线：

```text
school_id + year + depart_id + special_code + degree_type
```

如果无匹配或匹配不唯一，最终 CSV 中分数线字段留空。这样可以避免把 `score_level` 门类线误填到具体专业方向。

### 6.3 学科评估

学科评估按学校和学科代码匹配：

1. 优先使用 `school_id + level2_code`。
2. 匹配不到时回退 `school_id + level1_code`。
3. 仍匹配不到则留空。

## 7. 限流策略

`planDetailV2` 默认单线程采集，未缓存详情请求间隔 2 秒。遇到接口返回 `code=0010` 且 `message=请求频繁,请稍后再试` 时，先短间隔重试；仍失败则等待 30 秒再试；超过配置上限后把该详情记为 `error`，字段留空并继续处理下一条。

日志会定期输出详情进度，包括总计划数、已处理数、成功数、失败数和剩余待成功数，便于后台运行时查看状态。

## 9. 验收检查

| 检查项 | 要求 |
| --- | --- |
| 代码语法 | Docker 内 `python -m compileall /workspace/src` 通过 |
| 聚合 JSON | 5 个接口域 JSON 和 `crawl_summary.json` 均生成 |
| 样本 CSV | 只包含方向级计划行，不包含独立学校/分数线/学科评估行 |
| 全量范围 | 学校数为 21，年份只包含 2024、2025、2026 |
| 去重 | 不重复 `(school_id, year, plan_id, research_area, exam_subject_clean)` |
| 详情字段 | `exam_book_clean`、`research_area_note`、`min_score` 等有则写入，无则留空 |
| 分数线 | 只合并精确匹配的专业线，不用门类线兜底 |
| S04 | `source_registry.py --dry-run` 能从最终 CSV 去重得到候选学校 |

## 10. 数据缺口与边界

| 数据 | 当前处理 |
| --- | --- |
| 推免人数、统考名额 | 掌上考研 V2 未稳定提供拆分，后续官网资料补充 |
| 拟录取名单、报录比 | 不在掌上考研 V2 主流程中，后续从学校官网 PDF/Excel 提取 |
| 参考书 | `planDetailV2` 有则写入，没有就留空 |
| 专业线缺失 | 不用门类线替代，避免推荐误判 |
| 学校无公开数据 | 写入 summary 或日志，作为数据质量缺口处理 |
