### 1.2 设计目标

1. 保证主流程能在三周实训周期内完成：资料登记、数据提取、清洗入库、分析、可视化、推荐、报告。
2. 保证数据链路可追溯：每条核心数据都能关联到来源文件或来源页面。
3. 保证推荐结果可解释：每条推荐都要说明分数差、数据年份、招生计划和风险原因。
4. 保证本地环境可复现：所有同学以 Windows、PowerShell、Docker Desktop 为主要运行环境。
5. 保证代码结构清晰：数据脚本、后端服务、前端页面、SQL、配置、测试和文档按职责分开。
6. 明确第一版 S03 数据采集目标：以 `https://www.kaoyan.cn/school-list/50-0-0` 为入口，采集重庆研招单位候选库，并扩展采集招生计划、历年分数线和学科评估等级。
7. 保证安全和合规：掌上考研公开接口数据作为第一版结构化数据来源和候选范围参考，推荐、图表和报告使用前仍需保留来源、质量等级和官网核验提示。

## 2. 总体架构设计

### 2.1 系统整体架构

系统采用“离线数据处理 + 在线查询推荐 + 大数据分析展示”的结构。第一版不做复杂微服务，采用 Flask 单体后端和独立数据脚本，降低小组协作和部署成本。

```text
公开资料来源
  -> 掌上考研公开接口采集（院校、招生计划、分数线、学科评估）
  -> 原始 JSON data/raw
  -> 标准化 CSV data/processed
  -> S04/S07 入库、来源登记和质量检查

官方资料来源
  -> 数据成员人工下载 PDF、Excel、网页公告
  -> 来源登记 source_documents
  -> 原始文件 data/raw
  -> Python 提取脚本
  -> 临时提取文件 data/staging
  -> Python 清洗脚本
  -> 标准化文件 data/processed
  -> MySQL 业务库
  -> Flask API
  -> 前端页面和 ECharts 图表
  -> 用户查询、推荐、AI 报告

MySQL 标准化数据 / processed 文件
  -> HDFS 目录
  -> Hive 外部表
  -> Spark 离线分析
  -> analysis_results 分析结果表
  -> 图表接口
  -> 可视化看板
```

### 2.2 技术架构

| 层级 | 技术 | 主要职责 |
| --- | --- | --- |
| 数据来源层 | 掌上考研公开接口、高校官网、学院官网、研招网 | 掌上考研用于采集候选院校、招生计划、历年分数线和学科评估；官网和研招网用于核验及补充推免、拟录取、报录比等缺失数据 |
| 文件存储层 | 本地 `data/` 目录 | 保存原始文件、临时提取结果、清洗结果和分析结果 |
| 数据处理层 | Python 3.12、Requests、pandas、pdfplumber、openpyxl、BeautifulSoup | 掌上考研接口采集、PDF、Excel、网页表格提取，字段清洗，异常数据输出 |
| 业务存储层 | MySQL 8.0 | 保存学校、专业、来源、分数线、招生计划、拟录取和推荐记录 |
| 缓存层 | Redis 5.0.14 | 缓存图表数据、推荐结果和报告短期结果，第一版可作为增强项 |
| 大数据存储层 | HDFS、Hive 4.0.1 | 保存标准化数据，支撑离线分析演示 |
| 大数据计算层 | Spark 3.5.1 | 计算趋势、热度、分数统计、招生计划波动等指标 |
| 后端服务层 | Flask、SQLAlchemy 或 PyMySQL | 提供 API、推荐规则、报告生成、统一异常处理 |
| 前端展示层 | HTML、CSS、JavaScript、ECharts | 查询表单、学校列表、专业趋势、推荐结果和报告展示 |
| 部署运行层 | Docker、Docker Compose、PowerShell | 本地统一启动和检查 MySQL、Redis、Hadoop、Hive、Spark、Python 容器 |

### 2.4 关键设计决策

1. 后端使用 Flask 单体应用，避免三周内引入复杂服务拆分。
2. 数据提取、清洗、入库、同步、分析都设计为可单独运行的脚本，便于分工和测试。
3. 推荐规则使用配置文件管理阈值，不把阈值写死在页面或接口中。
4. MySQL 用于在线查询，Hive 和 Spark 用于离线分析和大数据能力展示。
5. 掌上考研公开接口可先提供候选库、招生计划、分数线和学科评估等结构化中间数据；其中推免人数、统考名额、拟录取名单和报录比仍需通过官网或研招网补充。
6. AI 报告必须基于系统查询出的结构化数据生成，大模型不可自行编造学校、专业和分数。

## 5. MySQL 数据库设计

### 5.1 数据库总体说明

数据库名称：`zhiyuan`。

字符集：`utf8mb4`。

核心设计原则：

1. 主键统一使用 `BIGINT AUTO_INCREMENT`。
2. 业务唯一性通过唯一索引约束，避免重复入库。
3. 所有核心表保留 `source_id`，保证数据可追溯。
4. 所有表保留 `created_at`、`updated_at`，必要时保留 `deleted_at`。
5. 状态字段使用英文枚举值，页面展示时再转换为中文。

## 6. 后端模块详细设计

### 6.1 掌上考研数据采集模块

模块文件：

| 文件 | 接口 | 数据域 | 输出目录 |
| --- | --- | --- | --- |
| `src/crawlers/school_list_crawler.py` | `/pc/school/schoolList` | 重庆 21 所研招单位 | `data/raw/kaoyan_v2/<batch_id>/school_list/school_list.json` |
| `src/crawlers/plan_list_v2_crawler.py` | `/pc/school/planListV2` | 近三年招生计划列表 | `data/raw/kaoyan_v2/<batch_id>/plan_list_v2/plan_list_v2.json` |
| `src/crawlers/plan_detail_v2_crawler.py` | `/pc/school/planDetailV2` | 专业详情、研究方向、参考书、近年分数 | `data/raw/kaoyan_v2/<batch_id>/plan_detail_v2/plan_detail_v2.json` |
| `src/crawlers/score_line_crawler.py` | `/pc/school/schoolScore` | 近三年分数线 | `data/raw/kaoyan_v2/<batch_id>/score_line/score_line.json` |
| `src/crawlers/level_rate_crawler.py` | `/pc/school/schoolLevelRate` | 学科评估 | `data/raw/kaoyan_v2/<batch_id>/level_rate/level_rate.json` |
| `src/crawlers/run_kaoyan_crawl.py` | 调度以上接口 | 方向级大合集 CSV 与 `crawl_summary.json` | `data/processed/kaoyan_v2_integrated/` |

目标入口：

```text
https://www.kaoyan.cn/school-list/50-0-0
```

主要职责：

1. 使用 `requests.post` 调用掌上考研公开接口，不使用浏览器自动化作为常规方案。
2. 按批次保存每次接口原始 JSON，保证接口变更后可回溯、可重解析。
3. 输出方向级大合集 CSV，作为 S04/S07 入库模块的中间数据。
4. 对接口失败、超时、字段缺失、单校单年无数据进行日志记录。
5. 支持同批次续跑：优先复用聚合 JSON，已成功详情不重复请求，失败详情可继续重试。
6. 同批次使用 `.crawl.lock` 加锁，防止多个进程同时覆盖聚合 JSON 和最终 CSV。
7. 在 `crawl_summary.json` 中输出 `batch_id`、产出路径、`plan_rows`、`unique_plan_count`、详情成功/失败数、分数线匹配数和学科评估匹配数。

字段映射摘要：

| 数据域 | 关键接口字段 | 中间字段或目标字段 | 说明 |
| --- | --- | --- | --- |
| 院校花名册 | `school_id`、`school_name`、`type_name`、`is_985`、`is_211`、`syl` | `candidate_school_id`、`university_name`、`school_type`、`school_level` | 写入 `universities` 前由 S04 处理 |
| 招生计划 | `depart_id`、`depart_name`、`special_code`、`special_name`、`degree_type`、`recruit_type_name`、`exam_subject`、`recruit_number` | `departments`、`majors`、`enrollment_plans` 的中间字段 | `recruit_number` 为专业招生总数，不含推免拆分 |
| 专业详情 | `plan_id`、`exam_book`、`research_area_data`、`level1_code`、`level2_code` | `planDetailV2` 按 `plan_id` 合并到方向级计划行，S07 可补充 `majors.exam_subjects` 或专业备注 | `planDetailV2` 有限流，默认 1 线程；未缓存请求全局间隔 2 秒，限流时先短间隔重试，仍限流则等待 30 秒 |
| 历年分数线 | `data_type`、`code`、`name`、`total`、`politics`、`english`、`special_one`、`special_two`、`diff_total` | `line_type`、`total_score_line`、单科线、`score_diff_to_national` | `score_level` 映射院校线，`school_score` 映射专业线 |
| 学科评估 | `code`、`code_name`、`rate`、`rate_sort`、`has_doctor` | `subject_code`、`subject_name`、`level_rate`、`rate_sort`、`has_doctor` | 按专业代码前 4 位与专业关联 |

实现约束：

1. 普通分页请求间隔默认 0.5 秒；`planDetailV2` 默认 1 线程，未缓存详情请求全局间隔 2 秒，避免高频访问。
2. 专业代码必须按字符串处理，避免前导零丢失。
3. `major_category` 中文门类可在 S06/S07 通过 `config/major_category_dict.yaml` 按代码前 2 位补齐；S03 CSV 至少保留 `major_category_code`。
4. `schoolScore.diff_total` 可作为反推国家线和推荐解释的辅助字段，但不能替代官网核验。
5. 掌上考研无法稳定提供推免人数、统考名额、拟录取名单和报录比；首版不做自动提取，这些字段由 S05 人工核验记录为数据缺口，后续版本再补充。

当前 S03 V2 完整性口径：

| 数据域 | 输出文件 | 验收口径 |
| --- | --- | --- |
| 院校花名册 | `school_list/school_list.json` | 全量批次应覆盖重庆 21 所 |
| 招生计划列表 | `plan_list_v2/plan_list_v2.json` | 按 `(school_id, year, plan_id)` 去重 |
| 专业详情 | `plan_detail_v2/plan_detail_v2.json` | 按 `plan_id` 去重，成功/失败数写入 summary |
| 分数线 | `score_line/score_line.json` | 最终 CSV 只合并精确匹配的专业线 |
| 学科评估 | `level_rate/level_rate.json` | 最终 CSV 按二级学科代码优先匹配，回退一级学科 |
| 最终 CSV | `kaoyan_v2_integrated_<batch_id>.csv` | 一行一个研究方向，不重复 `(school_id, year, plan_id, research_area, exam_subject_clean)` |

说明：缺失学校或字段是接口无返回或暂无对应业务数据，应进入数据质量记录。推荐、图表和报告展示时必须提示相关学校或专业数据不足。


### 6.2 来源登记模块

模块文件：`src/data_pipeline/source_registry.py`

主要职责：

1. 接收掌上考研 S03 V2 大合集 CSV，并按 `school_id + school_name` 去重得到候选招生单位数据。
2. 维护 21 个重庆研招单位候选库及官网核验状态。
3. 登记掌上考研接口批次、官方 PDF、Excel、网页公告等来源。
4. 记录来源资料处理状态。
5. 为后续提取、清洗、入库提供 `source_id`。

核心函数设计：

```python
def run_registry(csv_path: str | None = None, dry_run: bool = False) -> dict:
    """导入统一 CSV 中的重庆研招单位候选库，返回批次摘要。"""

def register_source_document(payload: dict) -> int:
    """登记一条官方资料来源，返回 source_id。"""

def update_source_status(source_id: int, status: str, remark: str | None = None) -> None:
    """更新来源资料处理状态。"""
```

处理规则：

1. 候选库来自掌上考研院校库重庆筛选结果，但字段入库时状态默认为 `pending`。
2. 官方资料必须填写 `source_url` 或 `local_path` 中至少一个。
3. 对学校名称无法匹配候选库的来源资料，先进入异常表，不直接入库。

### 6.3 官方资料人工核验模块

模块定位：

S05 不再作为首版自动 PDF/Excel/网页提取开发任务。当前系统以掌上考研 V2 大合集 CSV 为主数据源，官方资料只用于人工抽查、质量说明和缺口登记。

输入：

1. S03 V2 大合集 CSV。
2. `source_documents` 中登记的候选来源和后续人工补充来源。
3. 少量官网页面、PDF 或研招网公开页面的人工查看结果。

输出：

1. `data/quality/manual_verification_notes.md`。
2. 需要人工确认或后续补充的字段清单。
3. 可写入 `data_quality_issues` 的缺失或不一致问题。

处理流程：

```text
读取 S03 V2 大合集统计
  -> 抽查重点学校官网/研招网页面
  -> 对比学校名称、专业、考试科目、分数线等关键字段
  -> 记录推免人数、统考名额、拟录取名单、报录比等缺口
  -> 输出人工核验记录
  -> S06/S07 将缺失字段保留为空或写入质量问题
```

处理规则：

1. 不要求首版自动解析 PDF、Excel 或网页表格。
2. 官网与爬虫字段不一致时，先记录到人工核验文件，不直接覆盖爬虫数据。
3. 首版推荐和图表应提示第三方数据与缺口字段，避免把缺失字段解释为 0。

| `clean_score_line.py` | 清洗复试分数线 |
| `clean_enrollment_plan.py` | 清洗招生计划 |
| `clean_admission_record.py` | 清洗拟录取名单 |

通用清洗规则：

1. 去除字段前后空格、换行、制表符。
2. 将中文全角数字转换为半角数字。
3. 分数字段统一转换为 `Decimal` 或 `float`。
4. 年份字段统一转换为 `int`。
5. 学校名称必须匹配 `universities.university_name`。
6. 专业代码保留原始字符串，不转为数字，避免前导零丢失。
7. 学位类型统一为 `academic` 或 `professional`。
8. 学习方式统一为 `full_time` 或 `part_time`。

输出：

1. 标准化 CSV：`data/processed/<domain>/yyyy_<domain>.csv`。
2. 清洗报告：记录原始数量、有效数量、异常数量、重复数量。
3. 异常数据：写入 `data/quality/` 和 `data_quality_issues`。

### 6.8 分数线评估模块

模块文件：`src/services/score_service.py`

#### 国家线数据来源

由于官方国家线获取困难，本项目采用**从专业线反推国家线**的方法：

**推导原理**：

掌上考研 V2 API 提供的 `score_diff_to_national` 字段表示专业线与国家线的差值：

```
score_diff_to_national = 专业线总分 - 对应国家线总分
```

因此可以反推：

```
国家线 = 专业线 - score_diff_to_national
```

**推导结果**：

- 生成 242 条国家线数据（2024-2026 年）
- 覆盖 573 个专业类别（学科门类 + 专业学位类别）
- 标准差为 0 的专业占比 >95%，推导结果高度一致
- 推导脚本：`scripts/derive_national_lines.py`
- 生成 SQL：`sql/mysql/004_derived_national_lines.sql`

**数据特点**：

推导出的国家线粒度比官方更细：

| 类型 | 官方国家线 | 推导的国家线 |
| --- | --- | --- |
| 粒度 | 13 个学科门类（哲学、经济学、法学等） | 242 个专业类别 |
| 示例 | 工学：265 分 | 计算机技术：273 分<br>电子信息：273 分<br>机械：273 分 |
| 优点 | 权威 | 更精确，直接匹配专业 |

#### 输入参数

| 字段 | 类型 | 必填 | 说明 |
| --- | --- | --- | --- |
| `target_year` | int | 是 | 目标年份 |
| `major_category` | string | 是 | 专业门类 |
| `major_name` | string | 否 | 专业名称 |
| `university_id` | int | 否 | 目标学校 |
| `total_score` | int | 是 | 初试总分 |
| `politics_score` | int | 否 | 政治 |
| `english_score` | int | 否 | 英语 |
| `subject_one_score` | int | 否 | 业务课一 |
| `subject_two_score` | int | 否 | 业务课二 |

#### 输出字段

| 字段 | 说明 |
| --- | --- |
| `overall_status` | `unsafe`、`warning`、`safe` |
| `data_available` | 是否有可用分数线 |
| `line_type` | 使用的分数线类型（`major`、`university`、`national`） |
| `reference_year` | 使用的参考年份 |
| `total_score_line` | 总分线 |
| `total_score_diff` | 用户总分减参考线 |
| `single_subject_results` | 单科评估结果 |
| `matched_line_count` | 匹配到的分数线数量 |
| `reference_strategy` | 参考策略（单一匹配或保守最高值） |
| `reference_line` | 使用的参考分数线详情 |
| `candidate_lines` | 候选分数线列表（前10条） |
| `warnings` | 风险提示 |

#### 评估规则

1. 优先使用专业线（`line_type = 'major'`）。
2. 专业线缺失时使用院校线（`line_type = 'university'`）。
3. 院校线缺失时使用国家线（`line_type = 'national'`）。
4. 匹配到多条分数线时，按**总分线最高值**保守评估。
5. 总分或任一已填写单科低于参考线，状态为 `unsafe`。
6. 总分高于参考线但差值小于配置阈值（默认10分），状态为 `warning`。
7. 单科接近分数线时必须给出提示。
8. 数据完全缺失时返回 `data_available: false`，状态为 `warning`。

### 6.9 推荐模块

模块文件：`src/services/recommendation_service.py`

推荐流程：

```text
校验用户输入
  -> 调用分数线评估
  -> 查询候选学校和专业
  -> 查询近三年分数线、招生计划、拟录取统计
  -> 计算推荐指标
  -> 按配置化规则计算综合分
  -> 划分冲刺、稳妥、保底
  -> 生成推荐理由和风险提示
  -> 写入 recommendation_logs
```

核心指标：

| 指标 | 计算方式 | 用途 |
| --- | --- | --- |
| `total_score_diff` | 用户总分 - 参考复试线 | 判断基本安全度 |
| `admission_avg_diff` | 用户总分 - 近年拟录取平均分 | 判断与真实录取水平的差距 |
| `admission_min_diff` | 用户总分 - 近年拟录取最低分 | 判断保底可能性 |
| `single_subject_safety` | 单科是否全部过线 | 判断单科风险 |
| `plan_stability_score` | 近三年招生计划波动 | 判断招生稳定性 |
| `data_quality_score` | 数据完整度评分 | 判断推荐可信度 |
| `school_level_score` | 学校层次加权 | 调整冲刺难度 |

推荐档位初始规则：

| 档位 | 规则 |
| --- | --- |
| 冲刺 | 已过参考线，拟录取平均分差在 `-10` 到 `10` 分之间，或学校层次较高且分数差不大 |
| 稳妥 | 总分明显高于参考线，单科安全，接近或高于拟录取平均分 |
| 保底 | 总分明显高于参考线和拟录取最低分，招生计划较稳定，数据质量较高 |

推荐结果结构：

```json
{
  "rank_type": "stable",
  "university_name": "重庆邮电大学",
  "department_name": "计算机科学与技术学院",
  "major_code": "085404",
  "major_name": "计算机技术",
  "reference_years": [2024, 2025, 2026],
  "score_line": 310,
  "admission_avg_score": 345.5,
  "plan_count": 80,
  "score_diff": 45,
  "reason": "总分高于近年复试线 45 分，接近拟录取平均分，单科均超过参考线。",
  "warnings": ["拟录取名单只收集到 2 年数据，结果仅供参考"]
}
```

## 10. 数据质量设计

### 10.1 质量检查规则

| 检查项 | 规则 | 处理方式 |
| --- | --- | --- |
| 学校名称 | 必须能匹配 `universities` | 不能匹配则进入异常表 |
| 年份 | 必须为 2000 到当前年份加 1 | 异常则拒绝入库 |
| 专业代码 | 保留字符串，不能为空 | 缺失则进入人工校对 |
| 分数 | 必须为数字，范围 0 到 500 | 异常则进入人工校对 |
| 单科分数 | 必须为数字，范围 0 到 150 | 异常则进入人工校对 |
| 招生人数 | 必须为非负整数 | 异常则进入人工校对 |
| 来源 ID | 核心数据必须有关联来源 | 缺失则拒绝入库 |
| 第三方字段 | 必须通过官网或研招网复核 | 未核验则不作为推荐依据 |

### 10.2 数据质量等级

| 等级 | 条件 |
| --- | --- |
| `high` | 近三年复试线、招生计划、拟录取数据都完整，且来源已核验 |
| `medium` | 至少有复试线和招生计划，拟录取数据部分缺失 |
| `low` | 只有复试线或招生计划，缺少关键历史数据 |

推荐模块必须把 `low` 数据质量的专业加入风险提示。

## 11. 日志、异常与安全设计

### 11.1 日志规范

所有模块日志必须包含：

1. `trace_id`
2. 模块名
3. 输入摘要
4. 执行结果
5. 异常原因
6. 执行耗时

数据任务日志示例：

```text
2026-06-16 12:00:00 | INFO | clean_kaoyan_v2 | trace_id=xxx | batch_id=20260616_full_v2 | 清洗完成 total=12000 success=11920 failed=80
```

### 11.2 异常分类

| 异常类 | 场景 | 前端提示 |
| --- | --- | --- |
| `ValidationError` | 参数缺失或格式错误 | 参数错误，请检查输入 |
| `DataNotFoundError` | 没有查询到学校或专业数据 | 暂无对应数据 |
| `DataQualityError` | 数据不足以推荐 | 数据不足，结果仅供参考 |
| `DatabaseError` | MySQL 查询失败 | 系统繁忙，请稍后重试 |
| `FileProcessError` | PDF、Excel 处理失败 | 文件处理失败，请人工校对 |
| `ReportGenerateError` | 大模型报告失败 | 已切换为模板报告 |

### 11.3 安全设计

1. `.env` 不提交仓库，只提交 `.env.example`。
2. 前端不展示数据库错误堆栈。
3. 接口不返回 API Key、数据库密码和宿主机敏感路径。
4. 拟录取名单只做统计分析，默认不展示个人明细。
5. 报告和页面必须展示“仅供参考，不代表录取承诺”。

## 14. 编码规范

### 14.1 Python 规范

1. 文件名使用 `snake_case`。
2. 函数名使用 `snake_case`。
3. 类名使用 `PascalCase`。
4. 配置项和数据库字段使用英文。
5. 注释和文档字符串尽量使用中文，专业英文术语可以保留。
6. 不在业务代码中写真实账号、密码、Token。

1. 表名使用复数英文名。

3. 时间字段统一使用 `created_at`、`updated_at`。

2. 接口请求统一封装。

## 16. 关键风险与设计应对

| 风险 | 设计应对 |
| --- | --- |
| 官方 PDF 格式不统一 | 首版不做自动提取，必要时人工核验并记录缺口 |
| 掌上考研接口变更或不可用 | 保存最近一次聚合 JSON 作为样例数据，爬虫失败时给出明确日志，并保留人工导入候选库的降级方案 |
| 掌上考研部分学校或年份无数据 | 在 S03 完整性记录中列出缺口，S05 人工核验记录说明，S06/S07 写入数据质量问题 |
| 掌上考研大合集 CSV 与唯一键冲突 | S07 入库前按学校、学院、专业、年份、学位类型、学习方式和研究方向去重或聚合，必要时调整唯一键 |
| 掌上考研院校库与官网不一致 | 候选库字段保留核验状态，核心数据必须官网复核 |
| 拟录取名单缺失 | 推荐规则支持数据质量等级和风险提示 |
| 数据量较小无法体现大数据 | 使用 HDFS、Hive、Spark 完成规范链路和趋势分析 |
| 大模型接口不可用 | 使用模板报告降级 |
| 同学环境不一致 | 统一 Docker Compose 和 PowerShell 脚本 |
| 推荐被误解为录取承诺 | 页面、接口、报告统一输出仅供参考声明 |
