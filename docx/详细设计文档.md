### 1.2 设计原则

1. 先保证主流程可运行：数据入库、位次查询、志愿推荐、图表展示、AI 报告。
2. 符合三周实训周期：采用 Flask 单体应用 + 离线数据脚本，避免过度拆分。
3. 数据链路可追溯：原始数据、清洗数据、入库数据、分析结果都保留来源和处理记录。
4. 推荐规则可解释：每条推荐结果必须能展示分数、位次、位次差、风险原因。
5. 配置外置：数据库、大模型接口、文件路径、推荐阈值不写死在业务代码中。
6. 安全合规：不采集非公开数据，不在代码仓库保存真实密码和 API Key。

## 2. 总体架构设计

### 2.1 系统架构

系统采用“离线数据处理 + 在线查询推荐”的结构。

```text
公开数据源
  -> Python 爬虫采集
  -> 原始数据文件 data/raw
  -> Python 数据清洗
  -> 清洗数据文件 data/processed
  -> MySQL 业务库
  -> Flask 后端 API
  -> ECharts 前端页面
  -> 用户查询、推荐、AI 报告

清洗数据文件 / MySQL
  -> HDFS
  -> Hive 表
  -> Spark 分析任务
  -> 分析结果表
  -> Flask 图表接口
  -> ECharts 可视化
```

### 2.2 技术架构

| 层级 | 技术 | 作用 |
| --- | --- | --- |
| 数据采集层 | Python、Requests、BeautifulSoup 或 Scrapy | 采集公开录取数据、一分一段表、院校信息 |
| 数据处理层 | Pandas、NumPy | 清洗、去重、标准化、分数位次处理 |
| 业务存储层 | MySQL 8.0 | 保存核心业务表和查询数据 |
| 缓存层 | Redis 5.0.14 | 缓存推荐结果、热门图表数据、AI 报告短期结果 |
| 大数据存储层 | HDFS、Hive 4.0.1 | 保存清洗后数据和分析中间表 |
| 大数据计算层 | Spark 3.5.1 | 离线统计趋势、热度、位次波动 |
| 后端服务层 | Flask、SQLAlchemy 或 PyMySQL | 提供 HTTP API、推荐逻辑、报告生成 |
| 前端展示层 | HTML、CSS、JavaScript、ECharts | 查询表单、推荐结果、图表展示 |
| AI 能力层 | 大模型 API 或本地模型接口 | 生成个性化志愿填报建议报告 |
| 部署运行层 | Docker、Docker Compose | 本地统一启动数据库、大数据组件和 Web 服务 |

### 2.3 设计选型说明

1. 后端采用 Flask：轻量、适合课程项目，接口开发成本低。
2. 前端采用 Flask 模板 + 原生 JavaScript + ECharts：避免引入复杂前端工程，便于小组协作和答辩演示。
3. 数据处理采用 Pandas + Spark：Pandas 负责清洗和小规模验证，Spark 负责体现大数据分析链路。
4. MySQL 作为在线查询库：响应推荐和图表接口。
5. Hive/HDFS 作为离线分析库：用于展示大数据平台能力，不直接承担在线查询压力。
6. Redis 可作为增强项：本期可先实现接口，再根据时间加入缓存。

### 4.3 `config/recommend_rules.yaml`

推荐规则必须配置化，便于后期根据真实数据调整。

```yaml
rank_rules:
  default:
    rush:
      min_rank_diff: -3000
      max_rank_diff: 0
    stable:
      min_rank_diff: 1
      max_rank_diff: 5000
    safe:
      min_rank_diff: 5001
      max_rank_diff: 20000
  high_score:
    score_min: 600
    rush:
      min_rank_diff: -1500
      max_rank_diff: 0
    stable:
      min_rank_diff: 1
      max_rank_diff: 3000
    safe:
      min_rank_diff: 3001
      max_rank_diff: 12000

risk:
  volatility_warning_threshold: 0.15
  min_year_count: 1
  recommended_year_count: 3
  recent_year_weight: 0.6
  average_rank_weight: 0.4
```

说明：

1. `min_year_count` 表示最少可参与推荐的有效历史年份数。为保证演示可用，本期允许 1 年有效数据参与推荐。
2. `recommended_year_count` 表示推荐质量较高的目标历史年份数。少于该数量时，推荐结果必须附加“历史数据较少，仅供参考”的提示。
3. 如果有效历史年份数为 0，该院校不能进入推荐列表。

### 4.4 配置读取规则

1. `.env` 用于敏感配置和运行环境差异。
2. `config.yaml` 用于普通运行参数。
3. `recommend_rules.yaml` 用于推荐算法阈值。
4. 代码中禁止硬编码数据库密码、大模型密钥、绝对文件路径。

## 5. MySQL 数据库设计

### 5.1 数据库命名

建议数据库名：`zhiyuan`

字符集：`utf8mb4`

排序规则：`utf8mb4_0900_ai_ci`

### 5.1.1 数据分层口径

为保证后续数据开发和排查口径统一，项目采用轻量数据分层：

| 分层 | 存储位置 | 说明 |
| --- | --- | --- |
| RAW | `data/raw/`、`/data/zhiyuan/raw/` | 原始采集数据，只追加，不直接修改 |
| ODS | `data/processed/`、Hive `ods_` 表 | 清洗后的明细数据，字段统一、类型统一 |
| FACT/DIM | MySQL `fact_`、`dim_` 表 | 在线查询使用的事实表和维度表 |
| ADS | MySQL/Hive `ads_` 表 | 面向图表、推荐和报告的分析结果 |
| LOG | MySQL `_log` 表、`logs/` 文件 | 查询日志、任务日志、报告生成日志 |

数据处理原则：

1. RAW 层保留原始字段和来源，不覆盖历史文件。
2. ODS 层只做清洗和标准化，不做业务推荐判断。
3. ADS 层可以包含统计指标、风险等级、热度分等业务加工结果。
4. 在线接口优先读取 MySQL 的 FACT/DIM/ADS 表，不直接读取 RAW 文件。

### 5.5 数据约束说明

1. `subject_type` 统一使用：`物理类`、`历史类`、`理科`、`文科`。
2. `batch` 统一使用：`本科批`、`专科批`、`特殊类型招生` 等。
3. `min_rank` 为空的数据不能参与推荐算法，但可用于图表或异常分析。
4. 推荐使用的主要排序依据是位次，不直接用分数跨年份比较。
5. MySQL 表设计以逻辑外键为主，不强制创建物理外键，避免批量导入时因维度数据顺序问题导致 ETL 失败。
6. `college_id` 可作为关联优化字段，但业务查询必须兼容仅有 `college_name` 的历史数据。
7. 所有 ETL 入库脚本必须支持幂等执行，重复导入同一批数据时应更新或跳过，不应产生重复事实记录。

#### 6.6.5 参考位次计算

考虑最近年份更重要，采用加权位次：

```text
reference_rank = latest_min_rank * recent_year_weight + avg_min_rank * average_rank_weight
```

默认：

```text
recent_year_weight = 0.6
average_rank_weight = 0.4
```

如果只有一年有效历史数据：

```text
reference_rank = latest_min_rank
recommend_warning = "该院校可用历史录取数据较少，推荐结果仅作弱参考"
```

如果院校缺少位次数据：

```text
不进入推荐列表，仅可进入数据不足提示
```

#### 6.6.6 位次差计算

```text
rank_diff = reference_rank - student_rank
```

说明：

1. `rank_diff > 0`：考生位次优于参考录取位次，风险较低。
2. `rank_diff = 0`：考生位次接近参考录取位次。
3. `rank_diff < 0`：考生位次低于参考录取位次，有冲刺风险。

#### 6.6.7 档位判断流程

```text
读取推荐阈值配置
按考生分数选择高分段规则或默认规则
计算 rank_diff
读取院校风险等级 risk_level
如果 risk_level = high 且原档位为 safe，则降为 stable
如果 risk_level = high 且原档位为 stable，则降为 rush
如果 rank_diff 落在 rush 区间，则加入冲
如果 rank_diff 落在 stable 区间，则加入稳
如果 rank_diff 落在 safe 区间，则加入保
其他数据不进入推荐列表
```

#### 6.6.8 推荐排序规则

| 档位 | 排序方式 |
| --- | --- |
| 冲 | 按 `rank_diff` 从大到小排序，优先推荐接近 0 的冲刺院校 |
| 稳 | 按 `abs(rank_diff)` 从小到大排序，优先推荐匹配度高的院校 |
| 保 | 按 `rank_diff` 从小到大排序，避免保底院校过低 |

同档位二级排序：

1. 风险等级低优先。
2. 年份数据更完整优先。
3. 院校基础信息完整优先。

#### 6.6.9 推荐伪代码

```text
function recommend(request):
    validate request
    rank_result = RankService.query_rank(request.year, request.province, request.subject_type, request.score)
    student_rank = rank_result.rank

    years = [request.year - 1, request.year - 2, request.year - 3]
    admissions = AdmissionRepository.list_recent_admissions(
        request.province, request.subject_type, request.batch, years
    )

    grouped = group admissions by college_name
    result = {rush: [], stable: [], safe: []}

    for each college_name, history in grouped:
        if valid rank count == 0:
            continue
        if valid rank count < recommended_year_count:
            add warning "该院校历史录取数据较少"

        reference = calculate_reference_rank(history)
        rank_diff = reference.reference_rank - student_rank
        risk = CollegeRiskRepository.find(college_name, request.province, request.subject_type, request.batch)
        level = classify_level(rank_diff, risk.risk_level, rules)

        if level is not None:
            item = build_recommend_item(college, history, reference, rank_diff, risk, level)
            result[level].append(item)

    sort each level
    limit each level
    save query log
    return response
```

#### 6.6.10 推荐降级策略

1. 若 `ads_college_risk` 未生成，则只按历史录取表计算位次差。

### 7.3 Spark 分析任务设计

#### 7.3.1 院校分数趋势任务

文件：`src/analysis/spark_score_trend.py`

输入表：`ods_college_admission`

输出表或文件：`ads_score_trend`

计算逻辑：

```text

统计 min_score、avg_score、max_score、plan_count
输出到 Hive ads_score_trend 或 MySQL ads_score_trend
```

#### 7.3.2 院校位次趋势任务

文件：`src/analysis/spark_rank_trend.py`

计算逻辑：

```text
按 college_name、province、subject_type、batch、year 聚合 min_rank
使用窗口函数计算上一年 min_rank
rank_diff_from_last_year = 当前年 min_rank - 上一年 min_rank
volatility_rate = abs(rank_diff_from_last_year) / 上一年 min_rank
输出位次趋势结果
```

#### 7.3.3 专业热度任务

文件：`src/analysis/spark_major_hot.py`

热度分计算建议：

```text
college_count_score = min_max_norm(college_count)
plan_count_score = min_max_norm(plan_count_total)
difficulty_score = min_max_norm(avg_min_score)
hot_score = (college_count_score * 0.4 + plan_count_score * 0.3 + difficulty_score * 0.3) * 100
```

归一化规则：

```text
min_max_norm(x) = (x - min_value) / (max_value - min_value)
```

说明：

1. `college_count_score` 表示开设该专业的院校数量热度。
2. `plan_count_score` 表示招生计划规模热度。
3. `difficulty_score` 表示录取难度热度，平均最低分越高说明竞争越激烈。
4. 当 `max_value = min_value` 时，归一化结果统一取 `0.5`，避免除零。
5. 专业热度算法本期用于展示，不作为录取概率判断。

#### 7.3.4 院校风险任务

文件：`src/analysis/spark_college_risk.py`

计算逻辑：

```text
按 college_name、province、subject_type、batch 分组
统计 year_count、latest_year、latest_min_rank、avg_min_rank、min_rank_best、min_rank_worst、rank_stddev
volatility_rate = rank_stddev / avg_min_rank
如果 volatility_rate >= 0.15，则 risk_level = high
如果 volatility_rate >= 0.08 且 < 0.15，则 risk_level = medium
否则 risk_level = low
```

本期建议同步方向：

```text

```

原因：

1. 清洗数据文件是 MySQL 和 Hive 的共同来源，便于排查。

### 19.3 推荐规则核心函数

```python
def classify_level(rank_diff: int, risk_level: str, rules: dict) -> str | None:
    if rules["rush"]["min_rank_diff"] <= rank_diff <= rules["rush"]["max_rank_diff"]:
        level = "rush"
    elif rules["stable"]["min_rank_diff"] <= rank_diff <= rules["stable"]["max_rank_diff"]:
        level = "stable"
    elif rules["safe"]["min_rank_diff"] <= rank_diff <= rules["safe"]["max_rank_diff"]:
        level = "safe"
    else:
        return None

    if risk_level == "high" and level == "safe":
        return "stable"
    if risk_level == "high" and level == "stable":
        return "rush"
    return level
```

### 19.4 位次查询 SQL

SELECT year, province, subject_type, score, same_score_count, cumulative_count
FROM fact_score_segment
WHERE year = :year
  AND province = :province
  AND subject_type = :subject_type
  AND score <= :score
ORDER BY score DESC
LIMIT 1;

### 19.5 推荐候选院校 SQL

```sql
SELECT
  a.year,
  a.province,
  a.subject_type,
  a.batch,
  a.college_name,
  a.min_score,
  a.min_rank,
  a.plan_count,
  c.city,
  c.school_type,
  c.education_level
FROM fact_college_admission a
LEFT JOIN dim_college c ON a.college_name = c.college_name
WHERE a.province = :province
  AND a.subject_type = :subject_type
  AND a.batch = :batch
  AND a.year IN (:year1, :year2, :year3)
  AND a.min_rank IS NOT NULL;
```
